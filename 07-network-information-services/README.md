## 1. Известно, что единственным идентификатором получателя электронной почты, в том числе в схеме с выделенным почтовым сервером, является адрес пользователя вида name@domain.com. Каким образом письмо находит путь к почтовому серверу, обслуживающему данного получателя?

Письмо находит путь к почтовому серверу получателя через **DNS‑запросы (MX‑записи)**.

### Пошаговый процесс доставки письма

1. **Отправитель вводит адрес** `name@domain.com` и отправляет письмо. Почтовый клиент передаёт его на **исходящий
   почтовый сервер**
   (обычно SMTP‑сервер провайдера или компании).

2. **Сервер отправителя анализирует домен получателя** (`domain.com`) и выполняет **DNS‑запрос** для поиска **MX‑записей
   **
   (Mail Exchanger) этого домена.

3. **DNS возвращает список MX‑записей**. Каждая запись содержит:
    * имя почтового сервера (например, `mail.domain.com`);
    * приоритет (число — чем меньше, тем выше приоритет).

   Пример MX‑записей для `domain.com`:
    ```
    domain.com.  3600  IN  MX  10  mail1.domain.com.
    domain.com.  3600  IN  MX  20  mail2.domain.com.
    ```

4. **Сервер отправителя выбирает почтовый сервер**:
    * сначала пытается подключиться к серверу с **наименьшим приоритетом** (в примере — `mail1.domain.com`, приоритет
      10);
    * если он недоступен, пробует сервер с более высоким приоритетом (в примере — `mail2.domain.com`, приоритет 20).

5. **Установление SMTP‑сессии**:
    * сервер отправителя устанавливает соединение по протоколу **SMTP** с выбранным почтовым сервером получателя;
    * передаёт ему письмо, указывая адрес получателя (`name@domain.com`).

6. **Сервер получателя проверяет адрес**:
    * убеждается, что домен `domain.com` обслуживается им;
    * проверяет, существует ли пользователь `name` в его базе данных;
    * если всё верно, принимает письмо и помещает его в **почтовый ящик** пользователя `name`.

7. **Получатель получает письмо**:
    * подключается к своему почтовому серверу по протоколам **POP3**, **IMAP** или через веб‑интерфейс;
    * видит новое письмо в папке «Входящие».

### Ключевые компоненты

* **DNS (Domain Name System)** — «телефонная книга» интернета: по домену (`domain.com`) находит IP‑адрес почтового
  сервера.
* **MX‑запись (Mail Exchanger)** — специальная DNS‑запись, указывающая, какой сервер принимает почту для домена.
* **SMTP (Simple Mail Transfer Protocol)** — протокол для передачи писем между почтовыми серверами.
* **POP3/IMAP** — протоколы для доступа пользователя к своим письмам на сервере.

### Краткий итог

Путь письма определяется так:

1. Из адреса `name@domain.com` выделяется домен (`domain.com`).
2. Через **DNS** запрашиваются **MX‑записи** домена.
3. Выбирается почтовый сервер с **наивысшим приоритетом**.
4. Письмо передаётся по **SMTP**.
5. Сервер получателя сохраняет письмо в ящике пользователя `name`.

## 5. Укажите, какие из следующих утверждений правильны:</br> а) веб-сервер находится постоянно в активном состоянии, прослушивая ТСР-порт 80;</br> б) веб-сервер переходит в активное состояние, получив запрос от клиента на ТСР-порт 80;</br> в) после отправки всех объектов клиенту сервер разрывает с ним ТСР-соединение;</br> г) веб-сервер не хранит информацию о том, какие страницы пользователь уже посетил;</br> д) получив запрос от клиента, веб-сервер отображает все объекты на экране.

Ответ а), в), г)

**а) Верно.** Веб‑сервер действительно постоянно находится в активном состоянии (в режиме ожидания) и «прослушивает» 
TCP‑порт 80 (для HTTP) или порт 443 (для HTTPS), чтобы принимать входящие запросы от клиентов.

**б) Неверно.** Веб‑сервер не запускается в ответ на запрос — он уже работает и ждёт подключения. Если бы сервер запускался
только после получения запроса, задержка перед обработкой была бы слишком большой.

**в) Верно.** После того как сервер отправил запрошенные данные (например, HTML‑страницу и связанные с ней ресурсы), он 
обычно разрывает TCP‑соединение — это стандартная практика для HTTP/1.0 и может применяться в HTTP/1.1 (если не используется
поддержка постоянных соединений — *keep‑alive*).

**г) Верно.** Сам веб‑сервер по умолчанию не отслеживает, какие страницы посещал пользователь. Состояние сессии и история
просмотров, если они нужны, реализуются через дополнительные механизмы: cookies, сессии на стороне сервера, токены и т.д.,
но это уже надстройки, а не базовая функция HTTP.

**д) Неверно.** Веб‑сервер не отображает объекты на экране пользователя — это задача браузера (клиента). Сервер лишь отправляет
запрошенные данные (HTML, изображения, скрипты и т.п.), а браузер их получает, обрабатывает и отображает.

## 6. За счёт чего в версии НТТР/2 достигается ускорение процесса построения веб-браузером веб-страницы по сравнению с НТТР 1.0 и 1.1?

В HTTP/2 ускорение достигается за счёт нескольких ключевых улучшений по сравнению с HTTP/1.0 и HTTP/1.1:

1. **Мультиплексирование.** В HTTP/1.x для каждого запроса (HTML, CSS, изображения и т.д.) требовалось отдельное TCP‑соединение
либо последовательная отправка запросов в рамках одного соединения — это создавало задержки (*head‑of‑line blocking*). 
В HTTP/2 множество запросов и ответов могут передаваться одновременно в рамках **одного TCP‑соединения**, без ожидания 
завершения предыдущих.

2. **Сжатие заголовков (HPACK).** Заголовки HTTP (например, `User‑Agent`, `Cookie`) часто повторяются и занимают много места.
В HTTP/2 используется алгоритм **HPACK** — он сжимает заголовки, сокращая объём передаваемых данных и время на их отправку.

3. **Server Push.** Сервер может **заранее отправить** браузеру ресурсы, которые тот, скорее всего, запросит 
(например, CSS и JS‑файлы для текущей страницы), не дожидаясь отдельного запроса. Это сокращает количество циклов «запрос‑ответ».

4. **Бинарный протокол.** В отличие от текстового формата HTTP/1.x, HTTP/2 использует **бинарную структуру** данных. Это
упрощает и ускоряет разбор сообщений на стороне клиента и сервера, снижает вероятность ошибок при парсинге.

**Краткий итог:** HTTP/2 строит страницу быстрее, потому что:
* отправляет много ресурсов параллельно через одно соединение (мультиплексирование);
* уменьшает объём данных за счёт сжатия заголовков;
* позволяет серверу заранее «подталкивать» нужные ресурсы (Server Push);
* работает с более компактным и быстрым бинарным форматом.

## 7. В чём состоят недостатки организации почтовой службы, при которой почтовый клиент и сервер установлены на компьютере у каждого абонента?

Недостатки такой организации почтовой службы:

1. **Высокие требования к ресурсам.** На каждом компьютере нужно достаточно мощности и дискового пространства для работы
сервера и хранения почты — это затратно и нецелесообразно для обычных пользователей.

2. **Необходимость постоянной доступности.** Чтобы получать письма, компьютер абонента должен быть **всегда включён и подключён к сети**.
Если он выключен или отключён от интернета, входящие письма не дойдут (или будут теряться).

3. **Сложности администрирования и поддержки.** Каждый пользователь вынужден самостоятельно:
* настраивать почтовый сервер;
* обновлять ПО;
* обеспечивать безопасность;
* делать резервные копии.
  Для большинства это технически сложно.

4. **Проблемы с безопасностью.** Личные компьютеры хуже защищены от:
* взлома;
* вирусов и вредоносного ПО;
* потери данных (из‑за сбоев жёсткого диска, перебоев питания и т.п.).
  Риск утечки или потери почты значительно выше, чем при использовании централизованных решений.

5. **Отсутствие централизованного управления.** Невозможно:
* применять единые политики безопасности;
* фильтровать спам и вирусы на уровне провайдера;
* организовать централизованное резервное копирование и восстановление данных.

6. **Ограниченная масштабируемость.** При росте объёма почты (много писем, большие вложения) пользователю придётся:
* наращивать дисковое пространство;
* модернизировать оборудование.
  Это неудобно и дорого.

7. **Проблемы совместимости и синхронизации.** Если пользователь хочет проверять почту с нескольких устройств (ПК, ноутбук, смартфон),
возникает сложность с синхронизацией:
* прочитанных/непрочитанных писем;
* отправленных сообщений;
* папок и структуры ящика.

8. **Нагрузка на сеть.** Каждый почтовый сервер должен напрямую обмениваться сообщениями с другими серверами. Это:
* увеличивает трафик на стороне пользователя;
* создаёт дополнительную нагрузку на домашнюю сеть и канал доступа в интернет.

**Краткий итог:** такая схема неэффективна из‑за высоких требований к оборудованию, необходимости постоянного подключения,
сложностей в настройке и поддержке, слабого уровня безопасности и проблем с доступом с разных устройств. Поэтому на практике
используют централизованные почтовые серверы (Gmail, Яндекс Почта и т.д.), а клиенты работают по протоколам SMTP/POP3/IMAP.

## 8. Агент системы управления сетью — это (выберите вариант ответа):</br>а) программа, устанавливаемая на управляемое устройство;</br>б) аппаратный блок, встраиваемый производителем в управляемое устройство;</br>в) аппаратный модуль, способный измерять характеристики проходящего через управляемое устройство трафика;</br>г) программа, способная по командам выполнять конфигурирование управляемого устройства;</br>д) программно-аппаратный модуль, управляющий вычислительным процессом в устройстве.

Ответ а), г)

**а) Программа, устанавливаемая на управляемое устройство** — верно. Агент системы управления сетью (Network Management Agent) — 
это программное обеспечение, которое развёртывается на управляемом устройстве (маршрутизаторе, сервере, коммутаторе и т.д.).
Его задача — собирать данные о состоянии устройства и передавать их системе управления (менеджеру).

**г) Программа, способная по командам выполнять конфигурирование управляемого устройства** — тоже верно. Агент не только
собирает и отправляет данные, но и может выполнять команды от системы управления: менять настройки, применять конфигурации,
запускать диагностические процедуры и т.п.

**Почему не остальные варианты:**

* **б)** Неверно. Агент — это в первую очередь **программа**, а не аппаратный блок. Производители могут встраивать поддержку
управления (например, чип для удалённого управления), но сам агент — ПО.
* **в)** Неверно. Описанный модуль больше похож на анализатор трафика или сенсор. Агент же отвечает за взаимодействие с 
системой управления, а не только за измерение трафика.
* **д)** Неверно. Формулировка слишком широкая и не специфична для агента. Программно‑аппаратный модуль, управляющий вычислительным
процессом, — это скорее часть ОС или аппаратной платформы, а не компонент системы управления сетью.

## 9. Что такое MIB в системе управления сетью?</br> Варианты ответов:</br>а) модель управляемого объекта;</br>б) база данных управляющей информации;</br>в) протокол взаимодействия агента и менеджера системы управления сетью;</br>г) набор характеристик объекта, необходимых для его контроля.

Ответ б)

MIB (Management Information Base, база данных управляющей информации) — это иерархическая виртуальная база данных, используемая
в системах сетевого управления (например, в SNMP — Simple Network Management Protocol). Она содержит:

* набор управляемых объектов (например, интерфейсы маршрутизатора, счётчики трафика, статусы портов);
* их параметры и текущие значения (количество переданных пакетов, загрузка процессора, состояние соединения и т.д.);
* структурированное описание этих объектов — как они называются, какого типа данные хранят, как их можно изменять.

Объекты в MIB организованы в виде дерева с уникальными идентификаторами (OID — Object Identifier), что позволяет агенту
и менеджеру точно ссылаться на нужный параметр.

**Почему не остальные варианты:**

* **а) модель управляемого объекта** — неверно. MIB описывает множество объектов, а не один объект. Модель отдельного объекта —
это лишь часть структуры MIB.
* **в) протокол взаимодействия агента и менеджера** — неверно. Протоколом взаимодействия является SNMP. MIB лишь определяет,
*какую информацию* можно запрашивать и изменять через этот протокол.
* **г) набор характеристик объекта, необходимых для его контроля** — частично похоже на правду, но слишком узко. MIB — это
не просто набор характеристик одного объекта, а централизованная база *всех* управляемых данных сети с чёткой иерархией
и идентификацией.

## 10. Какую роль играет агент в двухзвенной схеме управления устройством?</br> Варианты ответов:</br> а) роль клиента;</br> б) роль сервера;</br> в) роль посредника.

Ответ б)

В двухзвенной схеме управления (менеджер ↔ агент) агент выполняет роль **сервера** по отношению к менеджеру системы управления.

1. **Агент запускается на управляемом устройстве** (маршрутизаторе, коммутаторе, сервере и т.д.) и постоянно «слушает» запросы — 
то есть работает в режиме ожидания входящих соединений, как типичный сервер.

2. **Менеджер (клиент) инициирует взаимодействие**: отправляет запросы на получение данных или выполнение команд.

3. **Агент отвечает на запросы менеджера**, выполняя одну из двух задач:
   * предоставляет запрашиваемую информацию (например, данные о загрузке процессора, объёме свободной памяти, счётчиках трафика из MIB);
   * выполняет команду по изменению конфигурации устройства (если это разрешено).

4. **Типичный сценарий работы**:
   * менеджер отправляет запрос: *«Пришли данные о загрузке CPU за последние 5 минут»*;
   * агент получает запрос, собирает нужные данные с устройства, формирует ответ и отправляет его менеджеру;
   * менеджер отображает данные администратору.

**Почему не остальные варианты:**

* **а) роль клиента** — неверно. Клиент инициирует запросы. В схеме SNMP именно менеджер выступает в роли клиента, а агент
ждёт запросов и на них отвечает.
* **в) роль посредника** — неверно. Посредник (proxy) передаёт запросы между двумя сторонами. Агент не просто пересылает
данные: он собирает их с локального устройства и может выполнять команды на нём. Его функция — не маршрутизация запросов,
а непосредственное взаимодействие с управляемым ресурсом.

## 11. Какой протокол может быть использован для взаимодействия менеджера и агента системы управления сетью?</br> Варианты ответов:</br> а) SNMP;</br> б) SMTP;</br> в) MIP;</br> г) FTP;</br> д) Telnet.

Ответ а), д)

**а) SNMP (Simple Network Management Protocol)** — это **основной специализированный протокол** для взаимодействия менеджера
и агента в системах управления сетями. Он предназначен именно для сбора данных и конфигурирования сетевых устройств 
(маршрутизаторов, коммутаторов и т.д.).

Как это работает:
* менеджер отправляет агенту запросы (например, «получить значение счётчика ошибок на порту»);
* агент отвечает, предоставляя запрошенную информацию из MIB (Management Information Base);
* через SNMP также можно удалённо изменять настройки устройств.

**д) Telnet** — может использоваться для **ручного управления** устройством, где установлен агент. Хотя он не является
специализированным протоколом сетевого управления, администратор может подключиться по Telnet к устройству и выполнить
команды конфигурирования, фактически управляя агентом через командную строку.

**Почему не остальные варианты:**

* **б) SMTP (Simple Mail Transfer Protocol)** — протокол передачи электронной почты. Не имеет отношения к управлению сетевыми
устройствами. Его задача — доставка писем между почтовыми серверами.
* **в) MIP** — такого общепринятого сетевого протокола нет (возможно, опечатка или путаница с Mobile IP или другими аббревиатурами).
В контексте управления сетями MIP не используется.
* **г) FTP (File Transfer Protocol)** — протокол для передачи файлов. Он не предназначен для мониторинга состояния устройств
или их удалённого конфигурирования в рамках системы управления сетью. С его помощью можно, например, загрузить конфигурацию
или лог‑файл, но не управлять параметрами устройства в реальном времени.
