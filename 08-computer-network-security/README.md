|                                      | Объекты | Субъекты |
|--------------------------------------|---------|----------|
| Пользователи                         |         |          |
| Устройства                           |         |          |
| Прикладные процессы                  |         |          |
| Файлы                                |         |          |
| Пропускная способность каналов связи |         |          |
| Сетевые сервисы                      |         |          |

Ответ:

|                                      | Объекты | Субъекты |
|--------------------------------------|---------|----------|
| Пользователи                         |         | ✓        |
| Устройства                           | ✓       |          |
| Прикладные процессы                  |         | ✓        |
| Файлы                                | ✓       |          |
| Пропускная способность каналов связи | ✓       |          |
| Сетевые сервисы                      | ✓       |          |

**Объяснения:**

* **Субъекты** — это активные сущности, которые **запрашивают доступ** и выполняют действия с ресурсами.
    * **Пользователи** — люди или учётные записи, инициирующие запросы на доступ.
    * **Прикладные процессы** — запущенные программы или сервисы, действующие от имени пользователя и запрашивающие доступ к ресурсам.

* **Объекты** — это пассивные ресурсы, к которым **запрашивается доступ**.
    * **Устройства** (компьютеры, принтеры, накопители и т.д.) — физические или логические ресурсы, используемые субъектами.
    * **Файлы** — данные на носителях, к которым пользователи или процессы хотят получить доступ (прочитать, изменить, удалить).
    * **Пропускная способность каналов связи** — сетевой ресурс, который распределяется между пользователями и процессами
  (например, ограничение скорости для отдельных субъектов).
    * **Сетевые сервисы** (веб‑сервер, файловый сервер, база данных и т.п.) — программные ресурсы, предоставляющие определённые
  функции; доступ к ним контролируется (кто может подключиться и что делать).

## 2. Используя приведённый далее список терминов, вставьте пропущенные слова в следующее предложение: «Пользователи получают … к ресурсам ИС в результате …, однако прежде им необходимо успешно пройти … и …». (Аутентификация, авторизация, идентификация, права доступа).

Ответ - пользователи получают права доступа к ресурсам ИС в результате авторизации, однако прежде им необходимо успешно пройти идентификацию и аутентификацию

**Идентификация** — процесс, когда пользователь сообщает системе, *кто он* (например, вводит логин или номер учётной записи). 
Система «узнаёт» имя субъекта, но ещё не подтверждает его подлинность.

**Аутентификация** — проверка подлинности пользователя: система убеждается, что он действительно тот, за кого себя выдаёт.
Для этого пользователь предоставляет доказательство — например, пароль, отпечаток пальца, токен. Пока аутентификация не пройдена,
доступ не предоставляется.

**Авторизация** — определение, *какие действия* пользователь может выполнять и *к каким ресурсам* у него есть доступ. 
Происходит *после* успешной идентификации и аутентификации. На этом этапе система сверяет права учётной записи с правилами
доступа.

**Права доступа** — конкретные разрешения (читать, писать, запускать, изменять и т.д.), которые пользователь получает в 
результате авторизации. Это итоговый результат: пользователь может работать с теми ресурсами и функциями, которые ему разрешены.


**Последовательность процессов:**
1. Идентификация (пользователь называет себя).
2. Аутентификация (система проверяет, что это действительно он).
3. Авторизация (система определяет, что ему можно делать).
4. Предоставление прав доступа (пользователь начинает работать в рамках разрешённых возможностей).

## 3. Вставьте пропущенные слова из списка: «Стало известно, что в Интернете уже появились …, направленные на использование … новой версии браузера. Реализация данной … может привести к …, которая нанесёт … нашему предприятию». (Атака, уязвимость, эксплойт, ущерб, угроза).

Ответ - стало известно, что в Интернете уже появились эксплойты, направленные на использование уязвимости новой версии браузера. Реализация данной угрозы может привести к атаке, которая нанесёт ущерб нашему предприятию

**Эксплойт** — это программа, фрагмент кода или последовательность команд, которые используют **уязвимость** в ПО для проведения
атаки. То есть это «инструмент» злоумышленника.

**Уязвимость** — слабое место или ошибка в программном обеспечении (в данном случае — в новой версии браузера), которое
может быть использовано для нарушения безопасности системы.

**Угроза** — потенциальная возможность нанесения вреда. В контексте ИБ это означает, что наличие эксплойта и уязвимости 
создаёт риск атаки. Фраза «реализация угрозы» означает, что потенциальная опасность переходит в активные действия.

**Атака** — целенаправленные действия злоумышленника с использованием эксплойта для эксплуатации уязвимости. Это уже не 
потенциальный риск, а реальное событие.

**Ущерб** — негативные последствия, наступившие в результате успешной атаки. Для предприятия это может быть: утечка данных,
остановка сервисов, финансовые потери, репутационный вред и т.д.

**Логика последовательности:**
1. Сначала в сети появляются **эксплойты** (инструменты для атаки).
2. Они нацелены на конкретную **уязвимость** (слабость в ПО).
3. Наличие эксплойтов и уязвимостей формирует **угрозу** (риск атаки).
4. При реализации угрозы происходит **атака** (активное действие злоумышленника).
5. Атака приводит к **ущербу** (реальным негативным последствиям).

## 4. Приведите примеры ситуаций, при которых обеспечивается конфиденциальность, но не гарантируется целостность данных.

Ситуация, когда **конфиденциальность есть**, а **целостность не гарантирована**, означает, что:
* злоумышленник не может прочитать данные (они зашифрованы или передаются по защищённому каналу);
* но может изменить, повредить или подменить их — и получатель этого не заметит.

**Примеры таких ситуаций:**

1. **Передача зашифрованной информации по незащищённому каналу**
  * **Конфиденциальность:** обеспечивается шифрованием — перехвативший трафик не сможет прочитать содержимое.
  * **Целостность не гарантируется:** если нет механизма проверки целостности (например, цифровой подписи или хеш‑суммы), злоумышленник может:
    * изменить зашифрованные данные «вслепую» (например, перевернуть отдельные биты);
    * подменить весь зашифрованный блок на ранее перехваченный;
    * повторить старую передачу (атака replay).
  * **Пример:** использование только симметричного шифрования (AES) без HMAC или других средств контроля целостности.

2. **Использование устаревшего или слабого протокола шифрования.**
  * **Конфиденциальность:** на первый взгляд есть — данные зашифрованы.
  * **Целостность под угрозой:** если протокол не включает проверку целостности или использует уязвимый алгоритм 
(например, CBC без дополнительной аутентификации), возможны атаки на модификацию шифротекста.
  * **Пример:** ранние версии SSL с некоторыми наборами шифров.

3. **Шифрование без аутентификации источника.**
  * **Конфиденциальность:** данные зашифрованы для получателя.
  * **Целостность и подлинность не гарантированы:** атакующий может подменить зашифрованное сообщение своим, также зашифрованным
на том же ключе (если ключ известен или украден). Получатель расшифрует его и примет за подлинное.
  * **Пример:** обмен сообщениями с общим паролем без цифровой подписи.

4. **Хранение зашифрованных данных на ненадёжном носителе.**
  * **Конфиденциальность:** зашифрованный файл нельзя прочитать без ключа.
  * **Целостность не гарантирована:** если носитель повреждён (сбой диска, вирус) или данные изменены злоумышленником,
расшифровка пройдёт, но результат будет искажён или бесполезен.
  * **Пример:** зашифрованный архив на флешке, которую случайно повредили — после расшифровки файлы внутри могут быть битыми.

5. **Передача данных через доверенный, но не защищённый от модификации канал.**
  * **Конфиденциальность:** канал закрытый (например, выделенная линия), перехватить сложно.
  * **Целостность не гарантирована:** помехи в линии, преднамеренное искажение сигнала или вмешательство могут изменить 
данные, а у сторон нет способа это обнаружить.
  * **Пример:** передача зашифрованного трафика по радиоканалу, подверженному помехам или глушению.

**Краткий итог:**
Конфиденциальность без целостности возникает, когда есть защита *от чтения*, но нет защиты *от изменения*. Для полной 
безопасности нужно сочетать шифрование (конфиденциальность) с механизмами проверки целостности и подлинности 
(хеши, подписи, HMAC).

## 5. Как называется свойство, которое характеризует систему, в которой документ всегда имеет достоверную информацию о его источнике (авторе)?

Ответ - аутентичность

**Аутентичность** (или подлинность) — это свойство информации, гарантирующее, что её источник (автор) соответствует заявленному.
Иными словами, получатель может быть уверен: документ действительно создан тем, за кого себя выдаёт.

Для подтверждения аутентичности используются специальные механизмы защиты:
* **Цифровая подпись.** Позволяет проверить, что документ не был изменён после подписания и действительно подписан конкретным
лицом/системой. Основана на криптографии с открытым ключом.
* **Сертификаты электронной подписи.** Выдаются доверенными центрами сертификации (CA) и подтверждают связь между открытым
ключом и конкретным субъектом (человеком, организацией, сервером).
* **Механизмы аутентификации.** Например, пароли, токены, биометрия — помогают удостовериться, что действие (создание 
или отправка документа) выполнено уполномоченным лицом.
* **Журналы аудита.** Фиксируют, кто и когда создал/изменил документ. Хотя это не криптографическая защита, такие журналы
помогают отследить источник в рамках внутренней системы.

## 6. Сколько секретных ключей требуется для переписки 50 человек «каждый с каждым», использующих алгоритм DES?

Ответ 1225

Алгоритм DES (Data Encryption Standard) — это **симметричный** алгоритм шифрования. Это значит, что для шифрования и расшифрования
используется **один и тот же секретный ключ**. Чтобы два человека могли безопасно обмениваться сообщениями, у них должен 
быть общий секретный ключ.

Нужно посчитать, сколько уникальных пар можно составить из 50 человек — каждому участнику нужна отдельная пара ключей с 
каждым из остальных.

Для расчёта используется формула числа сочетаний без повторений (количество способов выбрать 2 человека из N):


C(n, 2) = (n * (n - 1)) / 2


Где:
* n — количество участников (в нашем случае 50);
* 2 — размер пары (переписка ведётся между двумя людьми).

Подставляем значения:

(50 * (50 - 1)) / 2 = (50 * 49) / 2 = 2450 / 2 = 1225 

**Пошагово:**
1. Первый участник должен иметь отдельные ключи со всеми остальными 49 участниками → 49 ключей.
2. Второй участник уже имеет ключ с первым, ему нужны ключи с оставшимися 48 участниками → 48 ключей.
3. Третий участник: ключ с первым и вторым уже есть, нужны ключи с 47 участниками → 47 ключей.
4. И так далее, пока не дойдём до последнего участника, которому не нужен ни один новый ключ (у него уже есть ключи со всеми).
5. Суммируем: 49 + 48 + 47 + ... + 1 = 1225$.

## 7. Что определяет криптостойкость алгоритма шифрования:</br> а) секретность алгоритма;</br> б) секретность ключа;</br> в) сложность алгоритма.

Ответ б)

Криптостойкость — это способность алгоритма противостоять попыткам взлома. Согласно **принципу Керкгоффса** (Kerckhoffs's principle),
безопасность криптосистемы должна зависеть исключительно от **секретности ключа**, а не от секретности алгоритма. Разберём подробнее:

1. **Секретность ключа (верный ответ)**:
  * В современных криптосистемах алгоритм обычно известен всем (и злоумышленнику в том числе).
  * Стойкость обеспечивается за счёт того, что ключ:
    * имеет достаточную длину (например, 128, 256 бит и более);
    * выбирается случайно и непредсказуемо;
    * надёжно хранится и передаётся.
  * Чтобы взломать шифр, злоумышленнику нужно перебрать огромное количество возможных ключей (полный перебор), что при 
достаточной длине ключа практически невозможно.

2. **Секретность алгоритма (неверно)**:
  * Полагаться на секретность алгоритма небезопасно:
    * его сложно сохранить в тайне (алгоритм может утечь, быть изучен через реверс‑инжиниринг);
    * при обнаружении уязвимости в секретном алгоритме его невозможно быстро заменить;
    * открытые алгоритмы проходят публичное тестирование и аудит, что повышает их надёжность.
  * Секретность алгоритма — это не гарантия стойкости, а скорее иллюзия безопасности («безопасность через неясность»).

3. **Сложность алгоритма (неверно)**:
  * Сложная структура алгоритма сама по себе не делает его стойким.
  * Если ключ слабый или короткий, даже сложный алгоритм можно взломать полным перебором ключей.
  * Чрезмерная сложность может привести к ошибкам в реализации и появлению уязвимостей.
  * Главное — не сложность, а математическая обоснованность стойкости и корректная реализация.

Криптостойкость определяется **секретностью и длиной ключа**. Современные стандарты (AES, RSA, ECC) спроектированы так,
чтобы оставаться стойкими даже при полной открытости алгоритма — при условии, что ключ остаётся секретным и достаточно длинным.

## 8. Можно ли передать секретный ключ по открытому каналу, не используя шифрование?

Ответ - да, с помощью алгоритма Диффи-Хеллмана

Алгоритм Диффи‑Хеллмана позволяет двум сторонам **выработать общий секретный ключ** по открытому (незащищённому) каналу 
связи, даже если все передаваемые сообщения могут быть перехвачены. При этом сам ключ **никогда не передаётся** в явном виде.

**Как это работает (упрощённо):**

1. Стороны (назовём их Алиса и Боб) заранее договариваются о двух **общедоступных параметрах**:
  * большое простое число p;
  * первообразный корень g по модулю p.
    Эти значения могут быть известны всем, включая потенциального злоумышленника.

2. Каждая сторона генерирует **свой секретный ключ**:
  * Алиса выбирает случайное число a и держит его в секрете;
  * Боб выбирает случайное число b и держит его в секрете.

3. Стороны вычисляют **открытые ключи** и обмениваются ими по открытому каналу:
  * Алиса вычисляет A = g^a mod p и отправляет Бобу;
  * Боб вычисляет $B = g^b mod p и отправляет Алисе.

4. Каждая сторона вычисляет **общий секретный ключ**, используя полученный открытый ключ и свой секретный ключ:
  * Алиса: K = B^a mod p = (g^b)^a mod p = g^ab mod p;
  * Боб: K = A^b mod p = (g^a)^b mod p = g^ab mod p.

5. В результате Алиса и Боб получают **одинаковый ключ** K, который можно использовать для симметричного шифрования (например, AES).


Злоумышленник, перехвативший p, g, A и B, не может легко вычислить K. Для этого ему нужно решить **задачу дискретного логарифмирования** — 
найти a или b по известным A или B. При достаточно больших значениях p эта задача считается вычислительно неразрешимой 
за разумное время.

* Алгоритм Диффи‑Хеллмана **не обеспечивает аутентификацию** сторон. Без дополнительных мер он уязвим к атаке «человек посередине»
(Man‑in‑the‑Middle): злоумышленник может подменить открытые ключи и установить отдельные ключи с Алисой и Бобом. Поэтому
DH обычно используют в сочетании с цифровыми подписями или сертификатами.
* На практике используются усовершенствованные версии алгоритма (например, ECDH — Elliptic Curve Diffie‑Hellman), которые
обеспечивают тот же уровень безопасности при меньших размерах ключей.

## 10. Какие из перечисленных ниже свойств образуют триаду безопасности CIA:</br> а) конфиденциальность;</br> б) неотказуемость;</br> в) аутентичность;</br> г) владение;</br> д) целостность;</br> е) полезность;</br> ж) захват привилегий;</br> з) доступность.

Ответ а), д), з)

1. **Конфиденциальность (Confidentiality)** — гарантия того, что информация доступна только тем, у кого есть соответствующие права доступа.
    * **Цель:** предотвратить несанкционированное ознакомление с данными.
    * **Примеры защиты:** шифрование данных, разграничение прав доступа, аутентификация.
    * **Пример нарушения:** утечка персональных данных из базы данных.

2. **Целостность (Integrity)** — гарантия того, что данные не были изменены или повреждены несанкционированно. Информация остаётся точной и достоверной.
    * **Цель:** защитить данные от несанкционированных изменений (случайных или преднамеренных).
    * **Примеры защиты:** контрольные суммы (хеши), цифровые подписи, журналы аудита.
    * **Пример нарушения:** злоумышленник изменил сумму в платёжном поручении.

3. **Доступность (Availability)** — гарантия того, что информация и ресурсы доступны авторизованным пользователям тогда, когда это необходимо.
    * **Цель:** обеспечить бесперебойную работу систем и доступ к данным.
    * **Примеры защиты:** резервное копирование, отказоустойчивость, защита от DDoS‑атак.
    * **Пример нарушения:** веб‑сайт компании недоступен из‑за DDoS‑атаки.

**Почему не остальные варианты:**

* **б) Неотказуемость (Non‑Repudiation)** — возможность доказать факт выполнения действия конкретным лицом, чтобы оно не
могло от него отказаться. Важное свойство, но не входит в триаду CIA.
* **в) Аутентичность (Authenticity)** — подтверждение подлинности субъекта или данных (например, проверка, что письмо 
действительно пришло от заявленного отправителя). Часто связана с аутентификацией и цифровой подписью. Входит в расширенные
модели безопасности, но не в базовую триаду.
* **г) Владение** — не является стандартным термином в информационной безопасности в контексте модели CIA.
* **е) Полезность** — субъективная характеристика данных, не относится к фундаментальным свойствам безопасности.
* **ж) Захват привилегий (Privilege Escalation)** — это тип атаки, при котором злоумышленник получает повышенные права доступа,
а не свойство безопасности.

Триада CIA — фундаментальная модель в информационной безопасности. Она задаёт три ключевых требования к защите информации:
* **C**onfidentiality (конфиденциальность) — «кто может читать»;
* **I**ntegrity (целостность) — «не изменено ли»;
* **A**vailability (доступность) — «можно ли получить сейчас».

## 11. Укажите, какие из перечисленных атак являются активными (выберите вариант ответа):</br> а) прослушивание сетевого трафика;</br> б) спуфинг;</br> в) социальный инжиниринг;</br> г) атака «отказ в обслуживании»;</br> д) внедрение вируса.

Ответ б), г), д)

**Активные атаки** предполагают вмешательство в работу сети или системы: злоумышленник не просто наблюдает, а изменяет,
прерывает или подменяет данные, воздействует на ресурсы. 

* **б) Спуфинг** (Spoofing) — злоумышленник маскируется под доверенный источник, подменяя IP‑адрес, MAC‑адрес, email и т.д.
Это прямое вмешательство: он активно выдаёт себя за другого, чтобы обмануть систему или пользователя.
* **г) Атака «отказ в обслуживании»** (DoS/DDoS) — злоумышленник перегружает ресурс (сервер, сеть) огромным количеством 
запросов, чтобы сделать его недоступным для легитимных пользователей. Действия целенаправленны и активны.
* **д) Внедрение вируса** — злоумышленник активно помещает вредоносное ПО в систему. Вирус может повреждать данные, красть
информацию, давать удалённый доступ и т.п. — это прямое воздействие на целевую систему.

**Почему не остальные варианты:**

* **а) Прослушивание сетевого трафика** (сниффинг) — **пассивная атака**. Злоумышленник просто перехватывает и анализирует
трафик, не вмешиваясь в его передачу и не изменяя данные.
* **в) Социальный инжиниринг** — метод манипуляции людьми для получения конфиденциальной информации (паролей, данных карт и т.д.).
Хотя он требует действий от злоумышленника, это скорее **психологическая тактика**, а не техническая атака на сеть или систему.
В классификации сетевых атак его обычно не относят к активным в том же смысле, что спуфинг или DoS.
* **е) Полезность** — не является видом атаки. Это характеристика данных или ресурса (насколько они ценны и применимы для пользователя).
* **ж) Захват привилегий** (Privilege Escalation) — это **результат** успешной атаки (например, после внедрения вируса 
или эксплуатации уязвимости), а не самостоятельный тип атаки. Сам по себе он не описывает действие злоумышленника в сети.
* **з) Доступность** — не атака, а одно из свойств информационной безопасности (часть триады CIA). Означает, что ресурсы 
должны быть доступны авторизованным пользователям.

## 12. Укажите, какие из следующих утверждений являются ошибочными:</br> а) Нет смысла предусматривать подсистему аутентификации приложения, основанную на использовании многоразовых паролей, если многоразовые пароли уже используются и при логическом входе в систему;</br> б) Надежная система авторизации может компенсировать недостаточную стойкость паролей пользователей, которые они использовали при входе;</br> в) Стойкость системы защиты считается достаточной, если затраты на нее превысили запланированный уровень;</br> г) Затраты на обеспечение безопасности информации, по крайней мере, не должны превышать величину потенциального ущерба от ее утраты;</br> д) Иногда стойкость системы обеспечения безопасности считают достаточной, если стоимость ее преодоления злоумышленниками превосходит стоимость полученной ими выгоды.

Ответ а), б), в)

**а) Ошибочное утверждение.**

* **Почему неверно:** даже если многоразовые пароли используются при входе в систему, дополнительная аутентификация на 
уровне приложения может усилить безопасность. Например:
    * злоумышленник, получивший доступ к учётной записи ОС, не получит автоматически доступ к приложению;
    * для особо чувствительных приложений (банковские системы, CRM с персональными данными) оправдан дополнительный барьер.
* **Вывод:** многоуровневая аутентификация повышает общую защищённость, поэтому смысл в такой подсистеме есть.

**б) Ошибочное утверждение.**

* **Почему неверно:** авторизация определяет, *что* пользователь может делать после входа, но не защищает от несанкционированного
входа. Если пароли слабые:
    * их легко подобрать (брутфорс);
    * они могут быть украдены (фишинг, утечки баз данных).
* **Пример:** даже самая строгая система авторизации не поможет, если злоумышленник вошёл в систему под учётной записью
администратора с простым паролем `12345`.
* **Вывод:** авторизация не компенсирует недостатки аутентификации — это разные этапы защиты.

**в) Ошибочное утверждение.**

* **Почему неверно:** затраты на защиту — лишь один из факторов. Достаточная стойкость определяется соотношением:
    * стоимости преодоления защиты для злоумышленника;
    * ценности защищаемой информации;
    * потенциального ущерба от её утечки/компрометации.
* **Пример:** тратить 1 млн руб. на защиту данных, ущерб от утечки которых — 10 тыс. руб., нерационально. И наоборот: защита
критически важных данных должна быть максимально надёжной, даже если это дорого.
* **Вывод:** критерий «затраты превысили запланированный уровень» не говорит о реальной стойкости системы.

**Почему остальные утверждения верны:**

**г) Верное утверждение.**

* Логика проста: если затраты на защиту превышают потенциальный ущерб, такая защита экономически нецелесообразна. Например,
нет смысла тратить миллионы на защиту общедоступной информации.

**д) Верное утверждение.**

* Это **основной экономический принцип информационной безопасности:** защита считается достаточной, если злоумышленнику 
невыгодно её взламывать. Если затраты на взлом (время, ресурсы, риск) выше, чем выгода от украденных данных, атака маловероятна.

## 13. Что из перечисленного может быть секретным ключом криптосистемы (выберите вариант ответа):</br> а) геометрическая фигура;</br> б) сборник детских сказок;</br> в) картина;</br> г) число 5.

Ответ а), б), в), г)

Секретный ключ в криптографии — это **набор данных произвольной природы**, который используется алгоритмом шифрования и 
дешифрования. Главное требование — ключ должен быть:
* известен только уполномоченным сторонам;
* достаточно сложен для угадывания/перебора;
* корректно обрабатываем алгоритмом шифрования.

Разберём каждый вариант:

**а) Геометрическая фигура**
* Может быть ключом, если её форма, размеры и координаты закодированы в числовую или символьную последовательность.
* Пример: координаты вершин многоугольника преобразуются в битовую строку — она и будет ключом.

**б) Сборник детских сказок**
* Весь текст книги или его часть (например, определённые страницы, строки) можно преобразовать в бинарную последовательность — получится длинный ключ.
* Важное условие: обе стороны должны точно знать, *какой именно* текст использовать (издание, страницы, абзацы).
* Пример: взять первые 1000 символов сборника и вычислить их хеш — результат будет ключом.

**в) Картина**
* Изображение можно представить как массив пикселей с числовыми значениями цвета (RGB). Эти данные преобразуются в длинную
битовую последовательность.
* Дополнительно можно использовать метаданные (размер, дата создания и т.д.).
* Пример: хеш от изображения JPEG-файла может служить ключом.

**г) Число 5**
* Простое число тоже может быть ключом — особенно в учебных примерах или упрощённых алгоритмах.
* В реальных системах используют очень большие числа (сотни и тысячи бит), но принцип тот же.
* Пример: в некоторых симметричных шифрах или в алгоритме Диффи‑Хеллмана небольшие числа демонстрируют логику работы,
хотя для безопасности нужны большие значения.

**Важные нюансы**

* **Формат данных.** Криптоалгоритмы работают с битовыми последовательностями. Любой объект (текст, изображение, фигура)
должен быть преобразован в такой формат.
* **Длина и энтропия.** Для реальной защиты ключ должен иметь достаточную длину и случайность (высокую энтропию). «Число 5»
или маленькая фигура слишком слабы против атак перебора.
* **Согласованность.** Обе стороны (отправитель и получатель) должны иметь *идентичную копию* ключа и понимать, как его
интерпретировать.

## 14. В каких из перечисленных ниже средствах обеспечения безопасности используется шифрование (выберите вариант ответа):</br> а) аутентификация;</br> б) авторизация;</br> в) протокол NAT;</br> г) защищенный канал;</br> д) сетевой экран прикладного уровня;</br> е) фильтрующий маршрутизатор;</br> ж) цифровая подпись.

Ответ а), г), ж)

* **а) Аутентификация.** Шифрование применяется для безопасной передачи учётных данных (логина и пароля) и защиты от перехвата. Например:
    * пароли могут храниться в виде хешей (с применением криптографических функций);
    * при двухфакторной аутентификации токены передаются в зашифрованном виде;
    * протоколы вроде Kerberos используют шифрование для обмена ключами и подтверждения подлинности.

* **г) Защищённый канал.** Шифрование — основа создания защищённых каналов связи. Примеры:
    * TLS/SSL — шифруют трафик между браузером и веб‑сервером;
    * IPsec — создаёт зашифрованные туннели в VPN;
    * цель — обеспечить конфиденциальность и целостность данных при передаче по открытым сетям.

* **ж) Цифровая подпись.** Шифрование (а точнее, криптография с открытым ключом) лежит в основе механизма цифровых подписей:
    * подпись создаётся с помощью приватного ключа отправителя;
    * проверяется с помощью публичного ключа;
    * гарантирует подлинность и целостность документа (если данные изменены, подпись станет недействительной).


**Почему остальные варианты не подходят:**

* **б) Авторизация.** Этот процесс определяет права доступа пользователя после успешной аутентификации. Шифрование здесь
**не используется напрямую** — система сверяет роли и разрешения, но не шифрует данные в рамках самой авторизации.

* **в) Протокол NAT (Network Address Translation).** NAT преобразует IP‑адреса пакетов для маршрутизации трафика между сетями.
Он **не выполняет шифрование** — его задача техническая (сопоставление адресов), а не безопасность.

* **д) Сетевой экран прикладного уровня (Application Layer Firewall).** Такие экраны анализируют трафик на уровне приложений (HTTP, FTP и т.д.),
фильтруют запросы по правилам, блокируют атаки. Но **шифрование не является их функцией** — они работают с уже расшифрованным
трафиком (или пропускают зашифрованный без изменений).

* **е) Фильтрующий маршрутизатор (Packet Filtering Router).** Фильтрует пакеты по IP‑адресам, портам и протоколам на сетевом
уровне (L3). **Не выполняет шифрование** — только разрешает или блокирует трафик на основе ACL (списков контроля доступа).

**Краткий итог:**

Шифрование используется там, где нужно:
* защитить данные от перехвата (**защищённый канал**);
* подтвердить подлинность сторон и безопасно передать учётные данные (**аутентификация**);
* гарантировать неизменность и авторство информации (**цифровая подпись**).

## 15. Какую роль в обеспечении безопасности компьютерной сети играет протокол NAT?

Ответ - скрывает внутреннюю структуру сети

**Как NAT (Network Address Translation) обеспечивает безопасность:**

1. **Маскировка внутренних IP‑адресов.**
    * NAT преобразует приватные (внутренние) IP‑адреса устройств в сети в один или несколько публичных (внешних) адресов.
    * Внешние узлы видят только публичный IP‑адрес маршрутизатора — они не знают, какие устройства и с какими адресами 
   находятся внутри сети.

2. **Ограничение инициации соединений извне.**
    * По умолчанию NAT не пропускает входящие пакеты, которые не являются ответом на запрос из внутренней сети.
    * Злоумышленник не может напрямую подключиться к компьютеру сотрудника (с приватным IP), так как этот адрес «невидим»
   из Интернета.

3. **Сокращение числа точек атаки.**
    * Вместо того чтобы «выставлять» наружу IP‑адреса всех компьютеров, принтеров и серверов, сеть «представлена» одним
   (или несколькими) публичными адресами.
    * Это уменьшает поверхность атаки: злоумышленнику сложнее найти уязвимое устройство.

4. **Естественная изоляция.**
    * Для доступа извне к какому‑либо внутреннему сервису (веб‑серверу, FTP) нужно явно настроить проброс портов (Port Forwarding).
   Без такой настройки внутренняя сеть остаётся «невидимой».

**Конкретные примеры защиты:**

* **Защита от сканирования портов.** Злоумышленник может просканировать публичный адрес роутера, но не увидит открытые
порты всех внутренних устройств — только те, что специально проброшены.
* **Предотвращение прямых атак.** Атака типа «перебор паролей» на компьютер пользователя из Интернета невозможна, если этот
компьютер не имеет прямого публичного IP и не настроен проброс порта для соответствующего сервиса.
* **Сокрытие топологии сети.** Злоумышленнику сложно понять, сколько устройств в сети, как они связаны и какие роли выполняют
(рабочие станции, серверы и т.д.).

**Важные нюансы:**

* NAT **не является полноценным средством защиты** — это в первую очередь механизм экономии IP‑адресов (IPv4).
* Он не заменяет:
    * файрволы (сетевые экраны), которые фильтруют трафик по сложным правилам;
    * системы обнаружения вторжений (IDS/IPS);
    * шифрование (VPN, TLS) для защиты содержимого трафика.
* Некоторые сценарии требуют отключения или обхода NAT (например, для пиринговых игр, VoIP без STUN/TURN), что может снижать уровень защиты.

Основная роль NAT в безопасности — **сокрытие внутренней структуры сети** и ограничение прямого доступа извне. Это создаёт
базовый «щит» за счёт маскировки приватных IP‑адресов и естественной блокировки несанкционированных входящих соединений. 
Однако для комплексной защиты его необходимо сочетать с другими средствами безопасности.

## 17. Установите соответствие следующих терминов на русском языке (ущерб; уязвимость; вредоносное ПО; доступность; целостность; подмена содержимого пакета; распределённая атака, «отказ в обслуживании») и английском языке (Denial of Service; availability; integrity; vulnerability; malware; impact; spoofing; DDoS).

Ответ:
* **ущерб** → **impact**;
* **уязвимость** → **vulnerability**;
* **вредоносное ПО** → **malware**;
* **доступность** → **availability**;
* **целостность** → **integrity**;
* **подмена содержимого пакета** → **spoofing**;
* **распределённая атака «отказ в обслуживании»** → **DDoS** (Distributed Denial of Service).

**Пояснения:**
1. **Ущерб (impact)** — негативные последствия от успешной кибератаки (финансовые потери, утечка данных, простой сервисов и т.д.).
2. **Уязвимость (vulnerability)** — слабое место в системе, ПО или конфигурации, которое может быть использовано злоумышленником
для проникновения или атаки.
3. **Вредоносное ПО (malware)** — общее название для программ, созданных с вредоносными целями: вирусы, трояны, шпионское ПО,
программы‑вымогатели и т.п.
4. **Доступность (availability)** — свойство системы или данных быть доступными для авторизованных пользователей тогда,
когда это необходимо. Входит в триаду безопасности CIA.
5. **Целостность (integrity)** — гарантия того, что данные не были несанкционированно изменены. Также часть триады CIA.
6. **Подмена содержимого пакета / подмена (spoofing)** — атака, при которой злоумышленник маскируется под доверенный источник,
подменяя IP‑адрес, MAC‑адрес, email и т.п., чтобы обмануть систему или пользователя.
7. **Распределённая атака «отказ в обслуживании» (DDoS / Distributed Denial of Service)** — атака на доступность ресурса,
при которой множество скомпрометированных устройств (ботнет) одновременно отправляют запросы к цели, перегружая её и делая
недоступной для легитимных пользователей.

## 18. Укажите, какие из нижеперечисленных свойств определяют одностороннюю функцию Y(x) (выберите вариант ответа):</br>а) Y(x) чрезвычайно сложно вычислить для любого входного значения х;</br>б) Y(x) просто вычисляется для любого входного значения х;</br>в) аргумент х легко вычислить по известному Y;</br>г) вычисление х по известному Y чрезвычайно затруднено;</br>д) вычисление х по известному Y абсолютно невозможно;</br>е) значения функции от близких значений аргументов не должны значительно отличать.

Ответ б), г)

**Односторонняя функция** (one‑way function) в криптографии — это математическая функция, которая:
1. **Легко вычисляется в прямом направлении** (свойство **б**): для любого заданного аргумента x можно быстро и эффективно
вычислить значение функции Y(x). Это необходимо для практического применения функции — например, при создании хешей паролей
или цифровых подписей.
2. **Практически необратима** (свойство **г**): зная значение Y, крайне сложно (вычислительно затратно) найти исходный 
аргумент x. «Чрезвычайно затруднено» означает, что для обратного вычисления потребуются нереалистичные ресурсы (время,
вычислительная мощность) — например, полный перебор всех возможных значений x.

**Почему не остальные варианты:**
* **а) Y(x) чрезвычайно сложно вычислить для любого входного значения х** — противоречит сути односторонней функции. Если
прямое вычисление сложно, функция бесполезна на практике (например, для хеширования).
* **в) аргумент х легко вычислить по известному Y** — прямо противоположно определению односторонней функции. В этом случае
функция была бы легко обратимой.
* **д) вычисление х по известному Y абсолютно невозможно** — слишком сильное утверждение. В теории, зная Y, всегда можно 
найти x полным перебором. На практике задача не «невозможна», а «вычислительно неосуществима за разумное время». Криптография
опирается на сложность, а не на абсолютную невозможность.
* **е) значения функции от близких значений аргументов не должны значительно отличаться** — относится к другим свойствам
функций (например, устойчивости к малым изменениям). Для односторонних функций, наоборот, важен **эффект лавины**: даже
небольшое изменение x должно приводить к сильному изменению Y(x) (это свойство криптографических хеш‑функций).

**Краткий итог:**

Односторонняя функция определяется двумя ключевыми свойствами:
* **лёгкость прямого вычисления** (x → Y(x));
* **трудность обратного вычисления** (Y(x) → x).

## 19. Поясните назначение алгоритма Диффи–Хеллмана (выберите вариант ответа):</br>а) решает проблему передачи секретного ключа по открытому каналу;</br>б) это то же самое, что и алгоритм RSA;</br>в) смягчает проблему масштабируемости распределения ключей.

Ответ а), в)

**а) Решает проблему передачи секретного ключа по открытому каналу**

Алгоритм Диффи‑Хеллмана (DH) позволяет двум сторонам **выработать общий секретный ключ** по незащищённому каналу связи, 
даже если все передаваемые сообщения могут быть перехвачены. При этом:
* сам секретный ключ **не передаётся** в явном виде;
* стороны обмениваются только открытыми значениями, из которых затем независимо вычисляют общий ключ;
* злоумышленник, перехвативший обмен, не может легко восстановить общий ключ (задача дискретного логарифмирования считается
вычислительно сложной).

**Пример применения:** используется на этапе установки защищённого соединения (например, в TLS) для согласования сеансового
ключа симметричного шифрования (AES и т.д.).


**в) Смягчает проблему масштабируемости распределения ключей**

В симметричных криптосистемах для связи «каждый с каждым» требуется большое число ключей: для n участников нужно (n * (n - 1)) / 2 ключей. 
DH помогает решить эту проблему:
* не нужно заранее распределять и хранить множество парных ключей;
* каждая пара сторон может в любой момент **динамически выработать общий ключ** по открытому каналу;
* это особенно важно в больших сетях, где управление ключами становится сложной задачей.

**Почему не вариант б):**

**б) Это то же самое, что и алгоритм RSA — неверно.**

Хотя оба алгоритма связаны с криптографией с открытым ключом, это **разные механизмы** с разными целями:

* **Диффи‑Хеллман** предназначен **только для выработки общего секретного ключа** по открытому каналу. Он не шифрует данные
напрямую и не подписывает сообщения.
* **RSA** — это **универсальная асимметричная криптосистема**, которая может:
    * шифровать данные;
    * создавать цифровые подписи;
    * также может использоваться для обмена ключами, но это лишь одна из его функций.
* **Математические основы разные:** DH опирается на сложность дискретного логарифмирования, RSA — на сложность факторизации
больших чисел.

## 20. Сколько ключей требуется для секретной переписки 50 человек «каждый с каждым», использующих алгоритм RSA?

Ответ 100. 50 открытых и 50 закрытых ключей

Алгоритм RSA — это **асимметричная криптосистема**, в которой каждый участник имеет **пару ключей**:
* **открытый ключ** (Public Key) — используется для шифрования сообщений и может быть доступен всем;
* **закрытый (приватный) ключ** (Private Key) — используется для расшифровки сообщений, должен храниться в секрете.

**Расчёт для 50 человек:**
* Каждый из 50 участников генерирует свою пару ключей: 1 открытый + 1 закрытый.
* Общее количество ключей: 50 * 2 = 100.
* Из них:
    * 50 открытых ключей — могут быть опубликованы в общем доступе (например, в каталоге);
    * 50 закрытых ключей — остаются у владельцев и не передаются никому.

**Как это работает на практике:**
1. Участник А хочет отправить зашифрованное сообщение участнику Б.
2. А находит **открытый ключ Б** (он общедоступен).
3. А шифрует сообщение с помощью открытого ключа Б.
4. Зашифрованное сообщение может расшифровать **только Б** — с помощью своего **закрытого ключа**, который есть только у него.

**Почему не меньше?**

В отличие от симметричных криптосистем (где для связи «каждый с каждым» требуется (n * (n - 1)) / 2 ключей), в асимметричной
схеме (как RSA) **не нужны парные ключи** для каждой пары пользователей. Каждому достаточно иметь одну собственную пару — и он
может безопасно обмениваться сообщениями с любым другим участником сети.

Для сравнения:
* В **симметричной криптографии** для 50 человек потребовалось бы (50 * (50 - 1)) / 2 = 1225 ключей — по отдельному секретному
ключу для каждой пары.
* В **асимметричной криптографии (RSA)** достаточно 100 ключей — масштабирование гораздо эффективнее.

## 21. Какие из следующих утверждений правильные (выберите вариант ответа):</br>а) открытый ключ нужно защищать от подглядывания;</br>б) открытый ключ нужно защищать от подмены;</br>в) теоретически возможно зашифровать текст одним ключом, а расшифровать другим, который абсолютно никак не связан с первым;</br>г) алгоритм RSA основан на использовании функции разложения произведения большого числа на сомножители.

Ответ б), г)

**б) Открытый ключ нужно защищать от подмены — верно**

* Открытый ключ по определению **доступен всем**, поэтому защищать его от «подглядывания» не требуется.
* Однако критически важно защитить его **от подмены** (атаки типа *Man‑in‑the‑Middle*). Если злоумышленник подменит открытый
ключ (например, выдаст свой ключ за ключ получателя), он сможет:
    * перехватывать и расшифровывать сообщения, адресованные жертве;
    * подписывать сообщения от имени жертвы.
* Для защиты от подмены используют **сертификаты открытых ключей** (PKI), выдаваемые доверенными удостоверяющими центрами (CA).

**г) Алгоритм RSA основан на функции разложения большого числа на сомножители — верно**

* В основе RSA лежит **математическая сложность факторизации** — разложения очень большого числа (обычно произведения двух
больших простых чисел) на исходные простые множители.
* **Принцип работы:**
    * Открытый и закрытый ключи генерируются на основе двух больших простых чисел p и q.
    * Их произведение n = p * q становится частью открытого ключа.
    * Злоумышленнику, чтобы вычислить закрытый ключ, нужно разложить n на p и q — при достаточно больших числах это практически
  невозможно за разумное время.

**Почему не остальные варианты:**

**а) Открытый ключ нужно защищать от подглядывания — неверно**
* Суть асимметричной криптографии в том, что **открытый ключ публикуется открыто** и предназначен для всеобщего использования.
Его доступность — необходимое условие работы системы (например, для шифрования сообщений или проверки подписей). Защита от
«подглядывания» здесь бессмысленна.

**в) Теоретически возможно зашифровать одним ключом, а расшифровать другим, никак не связанным с первым — неверно**

* В асимметричных системах (RSA и др.) ключи **математически связаны** между собой, хотя и не идентичны:
    * сообщение, зашифрованное открытым ключом, расшифровывается **только** соответствующим закрытым ключом;
    * сообщение, подписанное закрытым ключом, проверяется **только** соответствующим открытым ключом.
* Если ключи «абсолютно никак не связаны», расшифровка или проверка подписи **невозможны**. Это нарушило бы саму логику
криптосистемы.

## 22. В чём состоит ручная синхронизация паролей? Варианты ответов:</br>а) администратор синхронизирует значения БД паролей на разных серверах сети;</br>б) ручная процедура приведения всех паролей пользователя к единому значению;</br>в) использование одного и того же пароля для доступа к разным по степени защищённости сервисам.

Ответ в)

Ручная синхронизация паролей — это **практика пользователя**, а не административная процедура. Она заключается в том, что
человек сознательно устанавливает **идентичный пароль** для нескольких учётных записей или сервисов — например, использует
один пароль для:
* почтового ящика;
* соцсетей;
* онлайн‑банка;
* рабочих систем и т.д.

**Почему это «ручная» синхронизация:**
* Пользователь сам вручную вводит один и тот же пароль при регистрации или смене пароля в разных сервисах.
* Никаких автоматических механизмов синхронизации (например, через централизованную систему управления паролями или SSO)
не задействовано.
* Обновление пароля тоже выполняется вручную — пользователь должен зайти в каждый сервис отдельно и поменять пароль.

**Почему не остальные варианты:**

**а) Администратор синхронизирует значения БД паролей на разных серверах сети** — это **автоматизированная или управляемая
синхронизация** на уровне инфраструктуры. Например:
* репликация Active Directory между контроллерами домена;
* синхронизация LDAP‑серверов;
* использование централизованных систем аутентификации.
  Это техническая задача администратора, а не «ручная синхронизация» в рассматриваемом смысле.

**б) Ручная процедура приведения всех паролей пользователя к единому значению** — формулировка близка к сути, но содержит
неточность:
* В реальной практике пользователь не «приводит к единому значению» уже существующие пароли — он **изначально задаёт одинаковые**
пароли в разных системах.
* Фраза подразумевает некую целенаправленную операцию по изменению существующих паролей, тогда как ручная синхронизация — это
скорее **подход к выбору паролей** с самого начала.
* Кроме того, вариант б) звучит как разовое действие, а ручная синхронизация — это **модель поведения** пользователя при
работе с паролями.


**Важные нюансы и риски**

Ручная синхронизация паролей **считается небезопасной практикой**, потому что:
* Компрометация одного пароля (например, из‑за утечки данных на некритичном сайте) ставит под угрозу **все** остальные
аккаунты пользователя.
* Злоумышленник, получивший пароль от одного сервиса, может попробовать использовать его в критически важных системах 
(банк, почта, корпоративные ресурсы).
* Нарушает принцип **разделения привилегий** и минимизации рисков.

**Краткий итог:**

Ручная синхронизация паролей (ответ **в**) — это когда пользователь **вручную использует один и тот же пароль** в разных
сервисах, не прибегая к автоматизированным системам управления паролями. Это простой, но опасный подход к управлению учётными
данными.

## 23. Поясните, что представляют собой аутентификаторы из разряда «что-то знаю» (выберите вариант ответа):</br>а) многоразовый пароль;</br>б) одноразовый пароль, сгенерированный устройством;</br>в) преобразование пользователем в соответствии с некоторым правилом символов, выводимых системой на экран, и использование их в качестве пароля;</br>г) информация о месте и времени встречи.

Ответ а), в), г)

**а) Многоразовый пароль — верно**

* Это классическая форма аутентификатора типа «что‑то знаю»: пользователь запоминает пароль и многократно использует его
для входа в систему.
* Примеры: пароль от электронной почты, учётной записи в соцсети, корпоративного аккаунта.
* Главный недостаток — если пароль станет известен злоумышленнику, тот сможет войти в систему неограниченное число раз.

**в) Преобразование символов по правилу — верно**
* Относится к категории «что‑то знаю», потому что пользователь должен знать правило преобразования.
* Как это работает: система выводит набор символов (цифр, букв), а пользователь применяет заранее оговорённое правило
(например, сдвигает буквы по алфавиту, добавляет фиксированное число к цифрам, меняет порядок символов).
* Результат преобразования используется как одноразовый или сеансовый пароль.
* Ключевой элемент — знание правила, которое остаётся неизменным и хранится в памяти пользователя.

* Пример: банк показывает на экране 4 цифры, пользователь прибавляет к каждой 3 и вводит результат.

**г) Информация о месте и времени встречи — верно**
* Хотя это не классический пароль, такая информация тоже относится к категории «что‑то знаю».
* Используется как секретный маркер доверия: только участники договорённости знают, где и когда состоится встреча (или 
какое действие нужно выполнить).
* В цифровом контексте может применяться как часть процедуры аутентификации или восстановления доступа (например, «секретный
вопрос»).
* Пример: «Место встречи — у фонтана в 18:00» или ответ на секретный вопрос «В каком городе вы провели детство?».

**Почему не вариант б):**

**б) Одноразовый пароль, сгенерированный устройством — неверно**
* Этот аутентификатор относится к категории **«что‑то имею»** (something you have), а не «что‑то знаю».
* Пользователь не генерирует пароль сам и не знает заранее его значение — он получает его от физического устройства 
(аппаратного токена, мобильного приложения, SMS).
* Безопасность здесь основана не на знании, а на владении устройством, способным сгенерировать корректный одноразовый код.
* Примеры: Google Authenticator, RSA SecurID, SMS‑коды подтверждения.

**Краткий итог:**

Аутентификаторы типа **«что‑то знаю»** (something you know) — это любая **информация, хранящаяся в памяти пользователя**,
которую он может воспроизвести при необходимости:
* стандартные многоразовые пароли (**а**);
* правила преобразования данных для получения пароля (**в**);
* секретные сведения (место, время, ответы на вопросы) (**г**).

Они противопоставляются другим факторам аутентификации:
* **«что‑то имею»** — токены, смарт‑карты, телефоны (**б**);
* **«кем являюсь»** — биометрия (отпечатки пальцев, распознавание лица).

## 24. Отметьте недостатки аутентификации с использованием многоразовых паролей (выберите вариант ответа):</br>а) возможность разгадывания пароля, если он слишком короткий;</br>б) необходимость передачи пароля по сети в открытом виде;</br>в) возможность подслушивания, подглядывания, «выманивания» пароля;</br>г) ручная синхронизация;</br>д) трудность запоминания надёжных паролей;</br>е) сложность реализации.

Ответ а), в), г), д)

**а) Возможность разгадывания пароля, если он слишком короткий — верно**

* Короткие пароли легко подобрать методом **брутфорса** (полного перебора всех вариантов) или **словарной атаки**.
* Например, пароль из 4 цифр имеет всего 10^4 = 10 000 комбинаций — его можно взломать за секунды.
* Для надёжности рекомендуют использовать пароли длиной не менее 8–12 символов с буквами разного регистра, цифрами и спецсимволами.

**в) Возможность подслушивания, подглядывания, «выманивания» пароля — верно**
* **Подслушивание (сниффинг)**: если пароль передаётся по сети без шифрования, его можно перехватить.
* **Подглядывание**: злоумышленник может увидеть пароль, когда пользователь его вводит (например, через плечо или с помощью камеры).
* **«Выманивание» (фишинг)**: пользователя обманом заставляют сообщить пароль (через поддельные сайты, письма, звонки).
* Поскольку пароль многоразовый, его компрометация даёт злоумышленнику **постоянный доступ** к учётной записи.

**г) Ручная синхронизация — верно**
* При использовании одного пароля на разных сервисах пользователь вынужден **вручную** вводить его везде.
* Если пароль нужно сменить (из‑за утечки или по политике безопасности), его придётся **вручную обновлять** на всех сервисах и устройствах.
* Это неудобно и повышает риск того, что пользователь пропустит какой‑то сервис или продолжит использовать скомпрометированный пароль.
* Кроме того, ручная синхронизация сама по себе опасна: если один сервис скомпрометирован, **все остальные аккаунты тоже под угрозой**.

**д) Трудность запоминания надёжных паролей — верно**
* Надёжные пароли (длинные, со случайными комбинациями букв, цифр, спецсимволов) **сложно запомнить**.
* Из‑за этого пользователи:
    * выбирают простые пароли (даты рождения, имена, слова из словаря) — что снижает безопасность;
    * записывают пароли на бумаге или в файле — что создаёт новые риски утечки;
    * используют один и тот же пароль для разных сервисов — что нарушает принцип разделения привилегий.

**Почему не остальные варианты:**

**б) Необходимость передачи пароля по сети в открытом виде — неверно**
* Современные системы **не передают пароли в открытом виде** по сети. Используются:
    * хеширование (на стороне клиента или сервера);
    * протоколы защищённой передачи (TLS/SSL);
    * механизмы аутентификации без передачи пароля (например, SCRAM, PAKE).
* Проблема не в самой передаче, а в том, что если пароль перехвачен (или украден иным способом), он остаётся действительным
для последующих входов.

**е) Сложность реализации — неверно**
* Аутентификация с многоразовыми паролями — **одна из самых простых** в реализации. Для неё:
    * не требуется специальное оборудование (в отличие от токенов или биометрии);
    * достаточно базы данных для хранения паролей (или их хешей);
    * легко интегрируется в любые системы.
* Именно простота реализации — одно из главных **преимуществ** парольной аутентификации, а не недостаток.

## 25. Укажите, что из перечисленного содержится в сертификате (выберите вариант ответа):</br>а) информация о владельце сертификата;</br>б) открытый ключ владельца сертификата;</br>в) закрытый ключ владельца сертификата;</br>г) открытый ключ сертифицирующей организации;</br>д) закрытый ключ сертифицирующей организации;</br>е) информация о сертифицирующем центре, выпустившем данный сертификат.

Ответ а), б), е)

**а) Информация о владельце сертификата — верно**

В сертификате указываются данные, позволяющие идентифицировать его владельца. Это может быть:
* имя физического лица;
* название организации;
* доменное имя (для SSL/TLS‑сертификатов);
* адрес электронной почты и т.д.

**б) Открытый ключ владельца сертификата — верно**

Это ключевой элемент сертификата. Открытый ключ:
* используется другими сторонами для шифрования данных, адресованных владельцу сертификата;
* применяется для проверки цифровых подписей, созданных владельцем с помощью его закрытого ключа;
* является общедоступным — его безопасно включать в сертификат.

**е) Информация о сертифицирующем центре, выпустившем сертификат — верно**
Сертификат содержит данные об удостоверяющем центре (Certificate Authority, CA), который его выдал. Обычно это:
* название CA;
* уникальный идентификатор;
* контактная информация;
* сведения о политике сертификации и т.п.
  Эта информация нужна для проверки подлинности сертификата — получатель может убедиться, что сертификат выдан доверенным центром.

**Почему не остальные варианты:**

**в) Закрытый ключ владельца сертификата — неверно**
* Закрытый ключ **никогда не включается** в сертификат и не передаётся никому.
* Он должен храниться в секрете у владельца и использоваться только им для:
    * расшифровки данных, зашифрованных открытым ключом;
    * создания цифровых подписей.
* Размещение закрытого ключа в сертификате полностью лишило бы смысла всю систему криптографической защиты.

**г) Открытый ключ сертифицирующей организации — неверно**
* Открытый ключ CA **не содержится непосредственно в сертификате владельца**.
* Для проверки подписи CA используется отдельный механизм: у проверяющей стороны должен быть заранее установлен и доверенный
открытый ключ самого CA (или корневого центра).
* В сертификате присутствует **подпись CA**, созданная с использованием его закрытого ключа, но не сам открытый ключ.

**д) Закрытый ключ сертифицирующей организации — неверно**
* Закрытый ключ CA — **строго конфиденциальная информация**.
* Он используется только внутри инфраструктуры CA для:
    * подписания (выпуска) сертификатов;
    * подписи списков отозванных сертификатов (CRL) и т.д.
* Компрометация закрытого ключа CA приведёт к компрометации всей цепочки доверия — поэтому он никогда не раскрывается и 
не включается ни в какие сертификаты.

## 26. Поясните, как убедиться в подлинности сертификата (выберите вариант ответа):</br>а) его надо расшифровать с помощью открытого ключа владельца сертификата и сравнить с открытой информацией сообщения;</br>б) его надо расшифровать с помощью закрытого ключа владельца сертификата и сравнить с открытой информацией сообщения;</br>в) его надо расшифровать с помощью открытого ключа сертифицирующей организации;</br>г) его надо послать для проверки в сертифицирующий центр.

Ответ в)

Процесс проверки подлинности сертификата (на примере стандарта X.509) состоит из нескольких шагов:

**Шаг 1. Получить открытый ключ удостоверяющего центра (CA)**

У проверяющей стороны (браузера, ОС, приложения) уже должен быть предустановлен и доверен открытый ключ CA (или корневого центра),
который выдал проверяемый сертификат.

**Шаг 2. Расшифровать цифровую подпись сертификата**
Сертификат содержит цифровую подпись, созданную CA. Для проверки:
* с помощью **открытого ключа CA** (полученного на шаге 1) расшифровывается подпись;
* в результате получается хеш‑сумма (дайджест) данных сертификата, которую рассчитал CA перед подписанием.

**Шаг 3. Самостоятельно вычислить хеш‑сумму данных сертификата**
Проверяющая сторона берёт открытую часть сертификата (данные владельца, его открытый ключ, срок действия и т.д.) и вычисляет
для неё хеш‑функцию (например, SHA‑256).

**Шаг 4. Сравнить хеши**
* Если хеш, полученный из расшифрованной подписи (шаг 2), совпадает с самостоятельно вычисленным хешем (шаг 3), сертификат **подлинный**:
    * не был изменён после подписания;
    * действительно подписан данным CA.
* Если хеши не совпадают, сертификат считается недействительным.

**в) Его надо расшифровать с помощью открытого ключа сертифицирующей организации — верно**
Это ключевой шаг проверки. Открытый ключ CA используется для расшифровки подписи и получения исходного хеша данных сертификата.

**а) Его надо расшифровать с помощью открытого ключа владельца сертификата… — неверно (некорректная формулировка)**
* Открытый ключ владельца нужен для других целей: шифрования данных для владельца или проверки его подписей.
* Он **не используется** для проверки подлинности самого сертификата.

**б) Его надо расшифровать с помощью закрытого ключа владельца сертификата… — неверно**
* Закрытый ключ владельца **строго секретен** и никогда не передаётся никому.
* Использовать его для проверки сертификата невозможно и бессмысленно.

**г) Его надо послать для проверки в сертифицирующий центр — неверно**
* Проверка сертификата выполняется **локально**, без обращения к CA.
* Для этого достаточно иметь доверенный открытый ключ CA.
* CA не занимается «онлайн‑проверкой» каждого сертификата — это сделало бы систему медленной и уязвимой.
* Для проверки статуса (не отозван ли сертификат) могут использоваться списки отозванных сертификатов (CRL) или протокол OCSP,
но это отдельный процесс.

## 27. Основная идея процедуры единого логического входа состоит в том, что (выберите вариант ответа):</br>а) пользователь выполняет логический вход в сеть только один раз при поступлении на работу, всё остальное время система выполняет только его авторизацию;</br>б) все пользователи выполняют процедуру логического входа с одного и того же компьютера;</br>в) пользователь выполняет логический вход в сеть только один раз, затем результат этой аутентификации используется другими серверами или приложениями.

Ответ в)

**Единый логический вход (Single Sign‑On, SSO)** — это технология, которая позволяет пользователю:
1. Пройти аутентификацию **один раз** (ввести логин и пароль).
2. Получить доступ к **разным сервисам и приложениям** без повторного ввода учётных данных.

**Как это работает:**
1. Пользователь вводит логин и пароль на странице входа (например, корпоративного портала или ID‑провайдера).
2. Система аутентифицирует пользователя и выдаёт ему **токен** (цифровой маркер доверия) или **билет** (например, в протоколах Kerberos).
3. При обращении к другому сервису (почте, CRM, файловому хранилищу и т.д.) система:
    * проверяет наличие действительного токена;
    * авторизует пользователя без запроса пароля.

**Примеры реализации:**
* **Kerberos** (используется в Active Directory): после входа в Windows‑систему пользователь получает билет, который позволяет
ему обращаться к сетевым ресурсам без повторной аутентификации.
* **SAML** (Security Assertion Markup Language): веб‑приложения доверяют аутентификации, проведённой централизованным 
провайдером (например, Google Workspace, Microsoft Entra ID).
* **OAuth 2.0 / OpenID Connect**: соцсети позволяют авторизоваться на сторонних сайтах через аккаунт Google, Facebook и т.п.

**Почему не остальные варианты:**

**а) Пользователь выполняет логический вход в сеть только один раз при поступлении на работу…** — неверно.
* Это **нереализуемо** на практике: сессии имеют ограниченный срок действия (обычно — до конца рабочего дня или после периода бездействия).
* Системы требуют периодического повторного входа для **безопасности** (защита от несанкционированного доступа, если пользователь
оставил компьютер без блокировки).
* Даже при длительном входе (например, «запомнить на 30 дней») система всё равно проводит **автоматическую аутентификацию**
по сохранённому токену/куки — это не «один раз на всю работу».

**б) Все пользователи выполняют процедуру логического входа с одного и того же компьютера** — неверно.
* SSO **не зависит от устройства или места входа**. Пользователь может:
    * войти с рабочего ПК, затем получить доступ к сервисам с планшета или телефона;
    * использовать разные компьютеры в офисе — аутентификация остаётся единой.
* Ключевое условие SSO — **централизованная проверка подлинности**, а не физическое место входа.

Основная идея единого логического входа (**SSO**) — **однократная аутентификация** с последующим **многократным использованием 
её результата** для доступа к разным ресурсам. Это:
* повышает удобство для пользователей (не нужно запоминать десятки паролей);
* упрощает администрирование (централизованное управление доступом);
* может усилить безопасность (меньше паролей == меньше рисков их компрометации).

## 28. Укажите цели использования электронной подписи (выберите вариант ответа):</br>а) доказательство целостности сообщения;</br>б) обеспечение конфиденциальности сообщения;</br>в) доказательство авторства сообщения.

Ответ а), в)

**а) Доказательство целостности сообщения — верно**

Электронная подпись гарантирует, что данные **не были изменены** после подписания. Как это работает:
1. Отправитель вычисляет **хеш‑сумму** (дайджест) исходного сообщения.
2. Хеш шифруется **закрытым ключом** отправителя — получается электронная подпись.
3. Получатель:
    * расшифровывает подпись с помощью **открытого ключа** отправителя (получает исходный хеш);
    * самостоятельно вычисляет хеш полученного сообщения;
    * сравнивает два хеша.
4. Если хеши совпадают, сообщение **не изменено**. Если нет — целостность нарушена.

**Пример:** если в подписанном договоре кто‑то изменил сумму сделки, электронная подпись станет недействительной.

**в) Доказательство авторства сообщения (аутентичность) — верно**

Электронная подпись подтверждает, что документ подписал **конкретный человек или организация**. Механизм:
* подпись создаётся **закрытым ключом**, который есть только у владельца;
* проверка выполняется **открытым ключом**, доступным всем;
* если подпись успешно проверена, значит, она точно создана владельцем закрытого ключа.
* это исключает возможность **отказа от авторства** («я этого не подписывал»).

**Пример:** подписанный отчёт сотрудника подтверждает, что его действительно подготовил и утвердил этот сотрудник.

**Почему не вариант б):**

**б) Обеспечение конфиденциальности сообщения — неверно**

Электронная подпись **не обеспечивает конфиденциальность** данных. Почему:
* сама по себе подпись **не шифрует** содержимое сообщения — оно остаётся читаемым для всех;
* цель подписи — подтвердить **целостность и авторство**, а не скрыть информацию;
* для конфиденциальности используют **шифрование** (например, с помощью открытого ключа получателя);
* на практике подпись и шифрование могут применяться **вместе**, но это разные механизмы:
    * сначала сообщение подписывают (подтверждают авторство и целостность);
    * затем шифруют (обеспечивают конфиденциальность).

## 29. Поясните, что из перечисленного характеризует дискреционный метод управления доступом, а что — мандатный (выберите вариант ответа):</br>а) описание прав доступа даётся в виде списков ACL;</br>б) право назначать права на доступ к объектам делегируются отдельным пользователям — владельцам объектов;</br>в) данная система управления доступом страдает от невозможности гарантированно проводить общую политику;</br>г) права доступа отдельного пользователя описываются ACE;</br>д) решение о предоставлении права доступа принимается операционной системой на основе правила;</br>е) субъект может по своему усмотрению передать часть своих полномочий другим субъектам;</br>ж) данная система управления доступом позволяет гарантированно проводить общую политику.

Ответ: дискреционный - а), б), в), г), е); мандатный - д), ж)

### Дискреционный метод управления доступом (DAC)
**а) Описание прав доступа даётся в виде списков ACL (Access Control List)**
* ACL — списки контроля доступа, где для каждого объекта (файла, папки) явно прописаны пользователи и их права (чтение, запись, выполнение).
* Пример: в Windows или Linux у каждого файла есть ACL, определяющий, кто и что может с ним делать.

**б) Право назначать права на доступ к объектам делегируются отдельным пользователям — владельцам объектов**
* Владелец объекта (например, создатель файла) сам решает, кому предоставить доступ и какие права дать.
* Это ключевая черта DAC: полномочия распределяются децентрализованно.

**в) Данная система управления доступом страдает от невозможности гарантированно проводить общую политику**
* Из‑за децентрализации владельцы могут устанавливать произвольные права, что затрудняет соблюдение единой политики безопасности в организации.
* Например, сотрудник может случайно или намеренно открыть доступ к конфиденциальному файлу всем пользователям.

**г) Права доступа отдельного пользователя описываются ACE (Access Control Entry)**
* ACE — элемент ACL, описывающий права конкретного субъекта (пользователя или группы) к объекту.
* Несколько ACE образуют полный ACL объекта. Это техническая деталь реализации DAC.

**е) Субъект может по своему усмотрению передать часть своих полномочий другим субъектам**
* Пользователь, обладающий правами на объект, может делегировать их другим пользователям.
* Такая гибкость — преимущество DAC, но и источник рисков (например, несанкционированное распространение доступа).

### Мандатный метод управления доступом (MAC)
**д) Решение о предоставлении права доступа принимается операционной системой на основе правила**
* Доступ регулируется централизованно, по строгим правилам, заданным системой безопасности.
* Учитываются метки безопасности: уровни секретности (например, «секретно», «совершенно секретно») и категории данных.
* Пользователь не может изменить правила доступа — даже владелец объекта не вправе дать доступ тому, у кого нет соответствующего уровня допуска.

**ж) Данная система управления доступом позволяет гарантированно проводить общую политику**
* MAC обеспечивает строгую, единообразную политику безопасности на всём предприятии.
* Правила доступа определяются централизованно (например, администратором безопасности) и применяются ко всем объектам и субъектам.
* Это критично для госорганов, военных и других структур с высокими требованиями к защите информации.

| Критерий                  | Дискреционный (DAC)                        | Мандатный (MAC)                           |
|---------------------------|--------------------------------------------|-------------------------------------------|
| **Кто управляет правами** | Владельцы объектов                         | Система (администратор безопасности)      |
| **Гибкость**              | Высокая (пользователь сам назначает права) | Низкая (жёсткие правила)                  |
| **Централизация**         | Децентрализованная                         | Централизованная                          |
| **Гарантии политики**     | Низкие (возможны отклонения)               | Высокие (строгое соблюдение)              |
| **Типичные примеры**      | Windows ACL, Linux POSIX permissions       | SELinux, Trusted Solaris, военные системы |

### 30. Основными функциями файервола являются (выберите вариант ответа):</br>а) аутентификация;</br>б) авторизация;</br>в) аудит;</br>г) фильтрация трафика.

Ответ в), г)

**в) Аудит — верно**

* Файервол (межсетевой экран) ведёт **журнал событий** (лог), фиксируя:
    * попытки доступа к сети (входящего и исходящего трафика);
    * заблокированные соединения (по правилам фильтрации);
    * подозрительную активность (например, сканирование портов);
    * успешные подключения.
* Эти данные нужны для:
    * анализа безопасности;
    * расследования инцидентов;
    * соответствия нормативным требованиям (например, PCI DSS, ФЗ‑152).

**г) Фильтрация трафика — верно**
* Это **основная функция** файервола. Он анализирует пакеты данных и принимает решение о пропуске или блокировке на основе заданных правил.
* При фильтрации проверяются:
    * IP‑адреса (источника и назначения);
    * порты;
    * протоколы (TCP, UDP, ICMP и т.д.);
    * состояние соединения (в stateful‑файерволах);
    * иногда — содержимое пакетов (в файерволах следующего поколения, NGFW).
* Цель — предотвратить несанкционированный доступ и атаки (DDoS, сканирование, эксплойты).

**Почему не остальные варианты:**

**а) Аутентификация — неверно**
* Аутентификация (проверка подлинности пользователя) **не является прямой функцией** файервола.
* Её выполняют другие системы:
    * службы каталогов (Active Directory, LDAP);
    * системы единого входа (SSO);
    * VPN‑шлюзы;
    * приложения с логином/паролем.
* Некоторые продвинутые файерволы могут интегрироваться с системами аутентификации (например, требовать логин перед доступом
к веб‑ресурсу), но это дополнительная, а не основная функция.

**б) Авторизация — неверно**
* Авторизация (определение прав доступа после аутентификации) тоже **не входит в базовый набор функций** файервола.
* Решения по авторизации принимают:
    * операционные системы (права на файлы/папки);
    * базы данных (права пользователей);
    * веб‑приложения и сервисы.
* Файервол может ограничивать доступ по IP/порту, но это **фильтрация на сетевом уровне**, а не полноценная авторизация 
с учётом ролей и привилегий пользователя.

## 31. Как может выглядеть запись в расширенном списке доступа, запрещающая передачу сообщений ICMP «Перенаправление маршрута» от любого хоста к хостам подсети 145.8.146.0/24?

Ответ - access-list 120 deny IMCP any 145.8.146.0 0.0.0.255 eq 5

### Разбор синтаксиса команды

1. **`access-list 120`**
* `access-list` — команда создания списка контроля доступа (ACL).
* `120` — номер ACL. В Cisco IOS:
    * номера 1–99 и 1300–1999 — для стандартных ACL (фильтруют только по IP‑адресу источника);
    * номера 100–199 и 2000–2699 — для расширенных ACL (могут фильтровать по IP, портам, протоколам и т.д.).
* Здесь используется номер 120 — это расширенный ACL, что позволяет задать сложные правила фильтрации.

2. **`deny`**
* Действие, которое будет применено к пакетам, соответствующим правилу.
* `deny` — запретить (отбросить) пакет.
* Альтернатива: `permit` — разрешить пакет.

3. **`icmp`**
* Протокол, который фильтруется. В данном случае — ICMP (Internet Control Message Protocol).
* ICMP используется для служебных сообщений: эхо‑запросы (ping), ошибки, перенаправления и т.п.

4. **`any`**
* Указывает источник трафика.
* `any` — «любой хост», то есть правило применяется к пакетам от всех IP‑адресов.
* Эквивалентно записи `0.0.0.0 255.255.255.255`.

5. **`145.8.146.0 0.0.0.255`**
* Целевая подсеть, к которой запрещаются пакеты.
* `145.8.146.0` — адрес подсети.
* `0.0.0.255` — wildcard mask (обратная маска), которая задаёт диапазон адресов:
    * В отличие от обычной маски подсети, wildcard показывает, какие биты **не важны**.
    * `0.0.0.255` означает: первые три октета должны совпадать точно (`145.8.146`), а последний может быть любым (`0–255`).
    * Таким образом, охватывается вся подсеть `145.8.146.0/24`.

6. **`eq 5`**
* Уточняет тип ICMP‑сообщения.
* `eq` (equal) — оператор сравнения, означает «равно».
* `5` — код типа ICMP‑сообщения «Перенаправление маршрута» (Redirect).
* Полный список типов ICMP можно найти в RFC 792. Код 5 соответствует сообщениям, которые маршрутизаторы отправляют хостам,
чтобы подсказать «лучший» маршрут.

### Почему именно так?
Правило блокирует ICMP‑пакеты типа «Перенаправление маршрута» (`eq 5`), идущие **от любого источника** (`any`) **к любому хосту**
в подсети `145.8.146.0/24` (`145.8.146.0 0.0.0.255`). Это нужно для:
* защиты от атак, где злоумышленник пытается изменить таблицу маршрутизации хостов, отправляя ложные ICMP Redirect;
* повышения безопасности сети — хосты не должны доверять перенаправлениям от произвольных маршрутизаторов.

### Важные нюансы
* **Порядок правил в ACL имеет значение.** Правила проверяются сверху вниз, до первого совпадения. Поэтому важно размещать
более специфические правила выше общих.
* **Неявное «deny any» в конце.** Если пакет не подходит ни под одно правило в ACL, он будет отброшен по умолчанию.
* **Применение ACL.** Сама по себе запись `access-list` не фильтрует трафик — её нужно привязать к интерфейсу командой 
`ip access-group 120 in` (или `out`) на нужном интерфейсе.

## 32. Что означает данный список доступа:</br>access-list 2 permit 93.8.25.2 0.0.0.0</br>access-list 2 deny 93.8.25.0 0.0.0.255?

Ответ - разрешить прохождение через маршрутизатор пакетов хоста 93.8.25.2, запрещая передачу пакетов, отправляемых любым
другим хостом подсети 93.8.25.0/24

### Разбор правил ACL
**Правило 1:**

```
access-list 2 permit 93.8.25.2 0.0.0.0
```

* `access-list 2` — создаётся стандартный список контроля доступа (ACL) с номером 2. Номера 1–99 и 1300–1999 относятся к 
стандартным ACL — они фильтруют трафик только по IP‑адресу источника.
* `permit` — действие: разрешить прохождение пакетов, соответствующих правилу.
* `93.8.25.2` — IP‑адрес источника, к которому применяется правило.
* `0.0.0.0` — wildcard mask (обратная маска). Означает, что **все четыре октета IP‑адреса должны точно совпадать**. То есть
правило применяется **только** к хосту `93.8.25.2`.

**Итог:** разрешить все пакеты, отправленные с хоста `93.8.25.2`.

**Правило 2:**

```
access-list 2 deny 93.8.25.0 0.0.0.255
```

* `deny` — действие: запретить прохождение пакетов, соответствующих правилу.
* `93.8.25.0` — базовый адрес подсети.
* `0.0.0.255` — wildcard mask. Означает:
    * первые три октета (`93.8.25`) должны совпадать точно;
    * последний октет может быть любым (от 0 до 255).
* Таким образом, правило охватывает **всю подсеть** `93.8.25.0/24` (адреса от `93.8.25.0` до `93.8.25.255`).

**Итог:** запретить все пакеты, отправленные из подсети `93.8.25.0/24`.

### Как работает этот ACL в целом
Важно помнить **порядок обработки правил**: маршрутизатор проверяет пакеты **последовательно**, сверху вниз, и применяет **первое подходящее** правило.


1. Пакет от хоста `93.8.25.2`:
    * совпадает с первым правилом (`permit`) → **разрешён**;
    * второе правило даже не проверяется.
2. Пакет от любого другого хоста в подсети `93.8.25.0/24` (например, `93.8.25.5`, `93.8.25.100`):
    * не подходит под первое правило (адрес не `93.8.25.2`);
    * подходит под второе правило (`deny`) → **запрещён**.
3. Пакет от хоста **вне** подсети `93.8.25.0/24` (например, `192.168.1.1`):
    * не подходит ни под одно из явных правил;
    * срабатывает **неявное правило** `deny any` (в конце любого ACL) → **запрещён**.

### Важные нюансы
* **Неявное `deny any`.** В конце каждого ACL всегда есть скрытое правило, запрещающее всё, что не разрешено явно. Поэтому
без дополнительных правил трафик извне подсети тоже будет заблокирован.
* **Привязка к интерфейсу.** Сама по себе запись ACL **не фильтрует трафик**. Чтобы правило заработало, его нужно применить
к интерфейсу маршрутизатора командой:

    ```
    ip access-group 2 in
    ```
  (или `out`, в зависимости от направления трафика).

* **Тип ACL.** Это **стандартный ACL** (номер 2), поэтому он фильтрует только по IP‑адресу источника. Для фильтрации по портам,
протоколам и т.д. нужны расширенные ACL (номера 100–199).
* **Безопасность.** Правило эффективно разрешает доступ только одному хосту (`93.8.25.2`) из всей подсети, блокируя остальных.
Это может использоваться, например, для:
    * ограничения доступа к критическому серверу;
    * изоляции тестового хоста;
    * создания «белого списка» доверенных устройств.

## 33. Какие из следующих утверждений о технологии NAT справедливы? Варианты ответов:</br>а) устройство, поддерживающее NAT, заменяет внешние IP-адреса пакетов, которые использовались при маршрутизации пакета через Интернет, внутренними;</br>б) устройство, поддерживающее NAT, фильтрует входящий трафик, отбрасывая все пакеты с частными IP-адресами назначения;</br>в) причина обращения к технологии NAT — дефицит адресов IPv4;</br>г) причина обращения к технологии NAT — скрытие адресов хостов для повышения безопасности сети.

Ответ а), в), г)

**NAT (Network Address Translation)** — это технология преобразования сетевых адресов. Проще говоря, она позволяет нескольким
устройствам в локальной сети (например, дома или в офисе) выходить в интернет через **один общий внешний IP‑адрес**.

**Как это работает:**
1. У каждого устройства в локальной сети есть свой **внутренний (частный) IP‑адрес** (например, `192.168.0.5`).
2. Роутер с функцией NAT заменяет внутренний IP‑адрес на **внешний (публичный) IP‑адрес** провайдера, когда устройство отправляет запрос в интернет.
3. Когда приходит ответ, роутер «помнит», какому устройству он предназначен, и передаёт данные нужному устройству по внутреннему адресу.

**Зачем нужен NAT:**
* **Экономия IP‑адресов.** IPv4‑адресов не хватает на все устройства, а NAT позволяет использовать один внешний адрес для десятков гаджетов.
* **Безопасность.** Внутренние IP‑адреса скрыты от интернета — злоумышленники не видят отдельные компьютеры или смартфоны в сети.
* **Удобство.** Можно подключать много устройств (ноутбуки, смартфоны, умные колонки и т.д.) без дополнительных настроек со стороны провайдера.

**Простой пример:**
Представьте многоквартирный дом с одним почтовым ящиком на входе. Письма (данные) приходят на общий адрес дома, а почтальон
(роутер с NAT) разносит их по квартирам (устройствам) — каждый получает своё.


**а) Устройство, поддерживающее NAT, заменяет внешние IP-адреса пакетов, которые использовались при маршрутизации пакета через Интернет, внутренними.**

Это верно. NAT (Network Address Translation) преобразует публичные (внешние) IP-адреса в частные (внутренние) при возврате
ответа в локальную сеть. Когда устройство из внутренней сети отправляет пакет во внешнюю сеть, маршрутизатор с NAT заменяет
внутренний IP-адрес устройства на публичный (глобальный). При получении ответа от внешнего сервера маршрутизатор, используя таблицу NAT,
заменяет публичный адрес обратно на внутренний, чтобы пакет был доставлен нужному устройству.

**в) Причина обращения к технологии NAT — дефицит адресов IPv4.**

Это основная причина появления и массового использования NAT. Технология была разработана в 1990-х годах в ответ на быстрое
исчерпание пула публичных IPv4-адресов. Из-за роста числа устройств и пользователей свободных адресов стало не хватать,
поэтому NAT позволил множеству устройств внутри локальной сети использовать один публичный IP-адрес для выхода в Интернет.

**г) Причина обращения к технологии NAT — скрытие адресов хостов для повышения безопасности сети.**

Это также верно. NAT повышает уровень безопасности сети, так как скрывает внутренние (частные) IP-адреса устройств от внешней сети.
Внешние устройства «видят» только публичный IP-адрес маршрутизатора, что усложняет несанкционированный доступ к внутренним 
ресурсам. Однако важно отметить, что NAT не обеспечивает полноценную защиту (например, не шифрует трафик и не аутентифицирует
пользователей), но создаёт дополнительный уровень конфиденциальности.

**б) Устройство, поддерживающее NAT, фильтрует входящий трафик, отбрасывая все пакеты с частными IP-адресами назначения.**

Это утверждение неверно. NAT сам по себе не является средством фильтрации трафика. Его основная задача — преобразование адресов,
а не блокировка пакетов. Фильтрация трафика обычно реализуется с помощью брандмауэров или других механизмов безопасности.
NAT может косвенно влиять на входящий трафик: пакеты, для которых нет соответствующей записи в таблице NAT (например, если
соединение не было инициировано изнутри сети), могут не пропускаться, но это связано не с частными IP-адресами как таковыми,
а с отсутствием трансляции.

## 34. Какова должна быть защита общедоступного сервера (компьютера-бастиона)? Варианты ответов:</br>а) поскольку доступ к этому компьютеру открыт, его помещают в демилитаризованную зону, поэтому он не требует защиты;</br>б) уровень защиты компьютера-бастиона такой же, как у серверов внутренней сети;</br>в) компьютер-бастион должен быть особенно тщательно защищён от внешних пользователей;</br>г) правила, определённые для компьютера-бастиона по доступу к ресурсам внутренней сети, должны быть более строгими, чем правила, регламентирующие доступ к нему внешних пользователей.

Ответ в), г)

**в) Компьютер‑бастион нужно тщательно защищать от внешних пользователей — верно**

Компьютер‑бастион (или «бастионный хост») — это сервер, который размещён в **демилитаризованной зоне (DMZ)** и напрямую 
доступен из интернета (например, веб‑сервер, почтовый сервер). **DMZ (демилитаризованная зона)** — это специальный сегмент сети,
который служит «буфером» между интернетом и внутренней (защищённой) сетью организации.

Проще говоря, это как **почтовый ящик у входа в дом**:
* почтальон (интернет‑пользователь) может положить в него посылку (данные) — например, отправить запрос на веб‑сайт;
* но он не может просто так зайти в сам дом (внутреннюю сеть), где хранятся ценные вещи (конфиденциальные данные).

**Зачем нужна DMZ:**
* защитить важные данные внутренней сети от атак из интернета;
* дать доступ к отдельным сервисам извне — но без прямого прохода вглубь инфраструктуры.

**Что размещают в DMZ:**
* веб‑серверы (сайты компании);
* почтовые серверы;
* FTP‑серверы;
* DNS‑серверы;
* другие сервисы, которые должны быть доступны из интернета.

**Как это работает:**
1. DMZ отделена от внутренней сети и от интернета с помощью **брандмауэров (файерволов)**.
2. Брандмауэр строго контролирует, какой трафик может:
    * попасть в DMZ из интернета (например, HTTP/HTTPS для веб‑сервера);
    * пройти из DMZ во внутреннюю сеть (обычно — только строго ограниченный набор соединений);
    * выйти из внутренней сети в DMZ.
3. Если злоумышленник взломает сервер в DMZ, он **не получит прямой доступ** к внутренней сети — брандмауэр не пропустит его дальше.

Поскольку компьютер‑бастион «на передовой» и открыт для внешних пользователей, его особенно важно защищать:
* установить и регулярно обновлять **антивирус**;
* настроить **файервол** с жёсткими правилами фильтрации;
* отключить ненужные службы и порты;
* регулярно устанавливать **обновления безопасности**;
* вести **аудит и мониторинг** событий.

Без усиленной защиты злоумышленники могут взломать этот сервер и использовать его как плацдарм для атаки на внутреннюю сеть.

**г) Правила доступа бастиона к внутренней сети должны быть строже, чем доступ к нему извне — верно**

Это принцип **минимальных привилегий**:
* **Доступ извне к бастиону** может быть относительно открыт (например, HTTP/HTTPS для веб‑сервера), чтобы пользователи могли им пользоваться.
* **Доступ бастиона внутрь корпоративной сети** должен быть максимально ограничен: он должен иметь доступ **только** к тем ресурсам,
которые строго необходимы для его работы.
* Например, веб‑сервер может обращаться к базе данных, но не должен иметь доступ к файловым серверам или рабочим станциям сотрудников.

Такой подход минимизирует риски: если бастион будет взломан, злоумышленник не сможет легко проникнуть вглубь сети.

**Почему не остальные варианты:**

**а) Поскольку доступ открыт, его помещают в DMZ, поэтому он не требует защиты — неверно**
* Размещение в DMZ **не отменяет необходимость защиты**. DMZ лишь изолирует сервер от внутренней сети, но не защищает сам
сервер от атак из интернета. Без защиты бастион легко взломать.

**б) Уровень защиты компьютера‑бастиона такой же, как у серверов внутренней сети — неверно**
* Серверы внутренней сети защищены периметровым файерволом и недоступны из интернета напрямую.
* Компьютер‑бастион, напротив, **открыт для всего интернета**, поэтому требует **более высокого уровня защиты**, чем обычные внутренние серверы.

## 35. В чём состоят главные отличия системы обнаружения вторжений (IDS) от файерволов с запоминанием состояния сеанса? Варианты ответов:</br>а) IDS не блокирует подозрительный трафик;</br>б) IDS анализирует не только события, связанные с прохождением трафика, но и другие подозрительные события в сети;</br>в) IDS не выполняет журнализацию событий;</br>г) в отличие от файерволов, системы IDS всегда являются исключительно программными.

Ответ а), б)

**а) IDS не блокирует подозрительный трафик — верно**

* **IDS (Intrusion Detection System, система обнаружения вторжений)** только **обнаруживает** подозрительную активность 
и **предупреждает** администратора (через логи, оповещения и т.д.).
* **Файервол с запоминанием состояния сеанса (stateful firewall)**, напротив, **активно блокирует** трафик, который не
соответствует правилам безопасности.
* То есть IDS — это «сторожевой пёс», который лает при виде незнакомца, а файервол — «охранник», который его не пускает.

**б) IDS анализирует не только трафик, но и другие события в сети — верно**
* IDS может отслеживать:
    * аномалии в поведении пользователей (например, попытки входа в необычное время);
    * попытки подбора паролей;
    * изменения в критически важных файлах (например, в системных библиотеках или конфигурациях);
    * признаки известных атак (по сигнатурам);
    * подозрительные сетевые паттерны (сканирование портов, DDoS‑атаки и т.п.).
* Файервол с stateful‑инспекцией фокусируется **только на сетевом трафике** и его соответствии правилам (IP, порты, состояние соединения).

**Почему не остальные варианты:**

**в) IDS не выполняет журнализацию событий — неверно**
* Наоборот, **ведение журналов (логирование)** — одна из ключевых функций IDS. Система:
    * записывает все подозрительные события;
    * сохраняет детали (IP‑адреса, время, тип атаки, затронутые ресурсы);
    * предоставляет данные для расследования инцидентов.
* Без журнализации IDS была бы бесполезна — администратор не смог бы понять, что и когда произошло.

**г) Системы IDS всегда являются исключительно программными — неверно**
* IDS бывают:
    * **программными** (например, Snort, Suricata — устанавливаются на сервер или рабочую станцию);
    * **аппаратными** (специализированные устройства с предустановленным ПО, интегрированные в сетевую инфраструктуру);
    * **гибридными** (программно‑аппаратные комплексы).
* Аналогично и файерволы: есть программные (Windows Defender Firewall) и аппаратные (Cisco ASA, FortiGate).

| Критерий               | IDS                                | Файервол (stateful)      |
|------------------------|------------------------------------|--------------------------|
| **Блокировка трафика** | Нет (только оповещение)            | Да (активная фильтрация) |
| **Область анализа**    | Трафик + события в системе/сети    | Только сетевой трафик    |
| **Журнализация**       | Да (подробные логи)                | Да (логи соединений)     |
| **Форма реализации**   | Программная, аппаратная, гибридная | Программная, аппаратная  |

## 36. Справедливо ли следующее утверждение: «Прокси-сервер всегда работает в режиме запоминания состояния сеанса»?

Ответ да

Прокси-серверы действительно функционируют с учётом состояния сеанса, что позволяет им анализировать и управлять трафиком
на прикладном уровне.

Прокси-сервер работает как посредник между клиентом и целевым сервером. Когда клиент отправляет запрос, прокси принимает его,
обрабатывает (например, проверяет соответствие политикам безопасности, модифицирует при необходимости) и затем пересылает
на целевой сервер от своего имени. При этом прокси-сервер «помнит» контекст взаимодействия — состояние сеанса, что позволяет ему:
* контролировать последовательность запросов и ответов в рамках одного сеанса;
* применять политики доступа, основанные на истории взаимодействия;
* поддерживать функции, такие как кэширование данных, где важно учитывать предыдущие запросы и ответы;
* обеспечивать корректную работу протоколов, требующих сохранения состояния (например, HTTP с использованием сессий или cookies).
* 
В отличие от простых межсетевых экранов, которые могут работать без учёта состояния (например, на сетевом уровне), прокси-серверы
оперируют на прикладном уровне модели OSI (L7) и анализируют не только сетевые и транспортные параметры (IP, порты), но и данные
прикладных протоколов (HTTP, HTTPS, FTP, SSH и др.). Это подразумевает отслеживание состояния сеансов этих протоколов, 
включая их начальные процедуры установления соединения и последующие обмены данными.

Таким образом, прокси-серверы по своей природе используют механизм запоминания состояния сеанса для выполнения своих функций — фильтрации трафика,
обеспечения анонимности, кэширования, балансировки нагрузки и других.
