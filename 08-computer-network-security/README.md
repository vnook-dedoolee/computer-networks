|                                      | Объекты | Субъекты |
|--------------------------------------|---------|----------|
| Пользователи                         |         |          |
| Устройства                           |         |          |
| Прикладные процессы                  |         |          |
| Файлы                                |         |          |
| Пропускная способность каналов связи |         |          |
| Сетевые сервисы                      |         |          |

Ответ:

|                                      | Объекты | Субъекты |
|--------------------------------------|---------|----------|
| Пользователи                         |         | ✓        |
| Устройства                           | ✓       |          |
| Прикладные процессы                  |         | ✓        |
| Файлы                                | ✓       |          |
| Пропускная способность каналов связи | ✓       |          |
| Сетевые сервисы                      | ✓       |          |

**Объяснения:**

* **Субъекты** — это активные сущности, которые **запрашивают доступ** и выполняют действия с ресурсами.
    * **Пользователи** — люди или учётные записи, инициирующие запросы на доступ.
    * **Прикладные процессы** — запущенные программы или сервисы, действующие от имени пользователя и запрашивающие доступ к ресурсам.

* **Объекты** — это пассивные ресурсы, к которым **запрашивается доступ**.
    * **Устройства** (компьютеры, принтеры, накопители и т.д.) — физические или логические ресурсы, используемые субъектами.
    * **Файлы** — данные на носителях, к которым пользователи или процессы хотят получить доступ (прочитать, изменить, удалить).
    * **Пропускная способность каналов связи** — сетевой ресурс, который распределяется между пользователями и процессами
  (например, ограничение скорости для отдельных субъектов).
    * **Сетевые сервисы** (веб‑сервер, файловый сервер, база данных и т.п.) — программные ресурсы, предоставляющие определённые
  функции; доступ к ним контролируется (кто может подключиться и что делать).

## 2. Используя приведённый далее список терминов, вставьте пропущенные слова в следующее предложение: «Пользователи получают … к ресурсам ИС в результате …, однако прежде им необходимо успешно пройти … и …». (Аутентификация, авторизация, идентификация, права доступа).

Ответ - пользователи получают права доступа к ресурсам ИС в результате авторизации, однако прежде им необходимо успешно пройти идентификацию и аутентификацию

**Идентификация** — процесс, когда пользователь сообщает системе, *кто он* (например, вводит логин или номер учётной записи). 
Система «узнаёт» имя субъекта, но ещё не подтверждает его подлинность.

**Аутентификация** — проверка подлинности пользователя: система убеждается, что он действительно тот, за кого себя выдаёт.
Для этого пользователь предоставляет доказательство — например, пароль, отпечаток пальца, токен. Пока аутентификация не пройдена,
доступ не предоставляется.

**Авторизация** — определение, *какие действия* пользователь может выполнять и *к каким ресурсам* у него есть доступ. 
Происходит *после* успешной идентификации и аутентификации. На этом этапе система сверяет права учётной записи с правилами
доступа.

**Права доступа** — конкретные разрешения (читать, писать, запускать, изменять и т.д.), которые пользователь получает в 
результате авторизации. Это итоговый результат: пользователь может работать с теми ресурсами и функциями, которые ему разрешены.


**Последовательность процессов:**
1. Идентификация (пользователь называет себя).
2. Аутентификация (система проверяет, что это действительно он).
3. Авторизация (система определяет, что ему можно делать).
4. Предоставление прав доступа (пользователь начинает работать в рамках разрешённых возможностей).

## 3. Вставьте пропущенные слова из списка: «Стало известно, что в Интернете уже появились …, направленные на использование … новой версии браузера. Реализация данной … может привести к …, которая нанесёт … нашему предприятию». (Атака, уязвимость, эксплойт, ущерб, угроза).

Ответ - стало известно, что в Интернете уже появились эксплойты, направленные на использование уязвимости новой версии браузера. Реализация данной угрозы может привести к атаке, которая нанесёт ущерб нашему предприятию

**Эксплойт** — это программа, фрагмент кода или последовательность команд, которые используют **уязвимость** в ПО для проведения
атаки. То есть это «инструмент» злоумышленника.

**Уязвимость** — слабое место или ошибка в программном обеспечении (в данном случае — в новой версии браузера), которое
может быть использовано для нарушения безопасности системы.

**Угроза** — потенциальная возможность нанесения вреда. В контексте ИБ это означает, что наличие эксплойта и уязвимости 
создаёт риск атаки. Фраза «реализация угрозы» означает, что потенциальная опасность переходит в активные действия.

**Атака** — целенаправленные действия злоумышленника с использованием эксплойта для эксплуатации уязвимости. Это уже не 
потенциальный риск, а реальное событие.

**Ущерб** — негативные последствия, наступившие в результате успешной атаки. Для предприятия это может быть: утечка данных,
остановка сервисов, финансовые потери, репутационный вред и т.д.

**Логика последовательности:**
1. Сначала в сети появляются **эксплойты** (инструменты для атаки).
2. Они нацелены на конкретную **уязвимость** (слабость в ПО).
3. Наличие эксплойтов и уязвимостей формирует **угрозу** (риск атаки).
4. При реализации угрозы происходит **атака** (активное действие злоумышленника).
5. Атака приводит к **ущербу** (реальным негативным последствиям).

## 4. Приведите примеры ситуаций, при которых обеспечивается конфиденциальность, но не гарантируется целостность данных.

Ситуация, когда **конфиденциальность есть**, а **целостность не гарантирована**, означает, что:
* злоумышленник не может прочитать данные (они зашифрованы или передаются по защищённому каналу);
* но может изменить, повредить или подменить их — и получатель этого не заметит.

**Примеры таких ситуаций:**

1. **Передача зашифрованной информации по незащищённому каналу**
  * **Конфиденциальность:** обеспечивается шифрованием — перехвативший трафик не сможет прочитать содержимое.
  * **Целостность не гарантируется:** если нет механизма проверки целостности (например, цифровой подписи или хеш‑суммы), злоумышленник может:
    * изменить зашифрованные данные «вслепую» (например, перевернуть отдельные биты);
    * подменить весь зашифрованный блок на ранее перехваченный;
    * повторить старую передачу (атака replay).
  * **Пример:** использование только симметричного шифрования (AES) без HMAC или других средств контроля целостности.

2. **Использование устаревшего или слабого протокола шифрования.**
  * **Конфиденциальность:** на первый взгляд есть — данные зашифрованы.
  * **Целостность под угрозой:** если протокол не включает проверку целостности или использует уязвимый алгоритм 
(например, CBC без дополнительной аутентификации), возможны атаки на модификацию шифротекста.
  * **Пример:** ранние версии SSL с некоторыми наборами шифров.

3. **Шифрование без аутентификации источника.**
  * **Конфиденциальность:** данные зашифрованы для получателя.
  * **Целостность и подлинность не гарантированы:** атакующий может подменить зашифрованное сообщение своим, также зашифрованным
на том же ключе (если ключ известен или украден). Получатель расшифрует его и примет за подлинное.
  * **Пример:** обмен сообщениями с общим паролем без цифровой подписи.

4. **Хранение зашифрованных данных на ненадёжном носителе.**
  * **Конфиденциальность:** зашифрованный файл нельзя прочитать без ключа.
  * **Целостность не гарантирована:** если носитель повреждён (сбой диска, вирус) или данные изменены злоумышленником,
расшифровка пройдёт, но результат будет искажён или бесполезен.
  * **Пример:** зашифрованный архив на флешке, которую случайно повредили — после расшифровки файлы внутри могут быть битыми.

5. **Передача данных через доверенный, но не защищённый от модификации канал.**
  * **Конфиденциальность:** канал закрытый (например, выделенная линия), перехватить сложно.
  * **Целостность не гарантирована:** помехи в линии, преднамеренное искажение сигнала или вмешательство могут изменить 
данные, а у сторон нет способа это обнаружить.
  * **Пример:** передача зашифрованного трафика по радиоканалу, подверженному помехам или глушению.

**Краткий итог:**
Конфиденциальность без целостности возникает, когда есть защита *от чтения*, но нет защиты *от изменения*. Для полной 
безопасности нужно сочетать шифрование (конфиденциальность) с механизмами проверки целостности и подлинности 
(хеши, подписи, HMAC).

## 5. Как называется свойство, которое характеризует систему, в которой документ всегда имеет достоверную информацию о его источнике (авторе)?

Ответ - аутентичность

**Аутентичность** (или подлинность) — это свойство информации, гарантирующее, что её источник (автор) соответствует заявленному.
Иными словами, получатель может быть уверен: документ действительно создан тем, за кого себя выдаёт.

Для подтверждения аутентичности используются специальные механизмы защиты:
* **Цифровая подпись.** Позволяет проверить, что документ не был изменён после подписания и действительно подписан конкретным
лицом/системой. Основана на криптографии с открытым ключом.
* **Сертификаты электронной подписи.** Выдаются доверенными центрами сертификации (CA) и подтверждают связь между открытым
ключом и конкретным субъектом (человеком, организацией, сервером).
* **Механизмы аутентификации.** Например, пароли, токены, биометрия — помогают удостовериться, что действие (создание 
или отправка документа) выполнено уполномоченным лицом.
* **Журналы аудита.** Фиксируют, кто и когда создал/изменил документ. Хотя это не криптографическая защита, такие журналы
помогают отследить источник в рамках внутренней системы.

## 6. Сколько секретных ключей требуется для переписки 50 человек «каждый с каждым», использующих алгоритм DES?

Ответ 1225

Алгоритм DES (Data Encryption Standard) — это **симметричный** алгоритм шифрования. Это значит, что для шифрования и расшифрования
используется **один и тот же секретный ключ**. Чтобы два человека могли безопасно обмениваться сообщениями, у них должен 
быть общий секретный ключ.

Нужно посчитать, сколько уникальных пар можно составить из 50 человек — каждому участнику нужна отдельная пара ключей с 
каждым из остальных.

Для расчёта используется формула числа сочетаний без повторений (количество способов выбрать 2 человека из N):


C(n, 2) = (n * (n - 1)) / 2


Где:
* n — количество участников (в нашем случае 50);
* 2 — размер пары (переписка ведётся между двумя людьми).

Подставляем значения:

(50 * (50 - 1)) / 2 = (50 * 49) / 2 = 2450 / 2 = 1225 

**Пошагово:**
1. Первый участник должен иметь отдельные ключи со всеми остальными 49 участниками → 49 ключей.
2. Второй участник уже имеет ключ с первым, ему нужны ключи с оставшимися 48 участниками → 48 ключей.
3. Третий участник: ключ с первым и вторым уже есть, нужны ключи с 47 участниками → 47 ключей.
4. И так далее, пока не дойдём до последнего участника, которому не нужен ни один новый ключ (у него уже есть ключи со всеми).
5. Суммируем: 49 + 48 + 47 + ... + 1 = 1225$.

## 7. Что определяет криптостойкость алгоритма шифрования:</br> а) секретность алгоритма;</br> б) секретность ключа;</br> в) сложность алгоритма.

Ответ б)

Криптостойкость — это способность алгоритма противостоять попыткам взлома. Согласно **принципу Керкгоффса** (Kerckhoffs's principle),
безопасность криптосистемы должна зависеть исключительно от **секретности ключа**, а не от секретности алгоритма. Разберём подробнее:

1. **Секретность ключа (верный ответ)**:
  * В современных криптосистемах алгоритм обычно известен всем (и злоумышленнику в том числе).
  * Стойкость обеспечивается за счёт того, что ключ:
    * имеет достаточную длину (например, 128, 256 бит и более);
    * выбирается случайно и непредсказуемо;
    * надёжно хранится и передаётся.
  * Чтобы взломать шифр, злоумышленнику нужно перебрать огромное количество возможных ключей (полный перебор), что при 
достаточной длине ключа практически невозможно.

2. **Секретность алгоритма (неверно)**:
  * Полагаться на секретность алгоритма небезопасно:
    * его сложно сохранить в тайне (алгоритм может утечь, быть изучен через реверс‑инжиниринг);
    * при обнаружении уязвимости в секретном алгоритме его невозможно быстро заменить;
    * открытые алгоритмы проходят публичное тестирование и аудит, что повышает их надёжность.
  * Секретность алгоритма — это не гарантия стойкости, а скорее иллюзия безопасности («безопасность через неясность»).

3. **Сложность алгоритма (неверно)**:
  * Сложная структура алгоритма сама по себе не делает его стойким.
  * Если ключ слабый или короткий, даже сложный алгоритм можно взломать полным перебором ключей.
  * Чрезмерная сложность может привести к ошибкам в реализации и появлению уязвимостей.
  * Главное — не сложность, а математическая обоснованность стойкости и корректная реализация.

Криптостойкость определяется **секретностью и длиной ключа**. Современные стандарты (AES, RSA, ECC) спроектированы так,
чтобы оставаться стойкими даже при полной открытости алгоритма — при условии, что ключ остаётся секретным и достаточно длинным.

## 8. Можно ли передать секретный ключ по открытому каналу, не используя шифрование?

Ответ - да, с помощью алгоритма Диффи-Хеллмана

Алгоритм Диффи‑Хеллмана позволяет двум сторонам **выработать общий секретный ключ** по открытому (незащищённому) каналу 
связи, даже если все передаваемые сообщения могут быть перехвачены. При этом сам ключ **никогда не передаётся** в явном виде.

**Как это работает (упрощённо):**

1. Стороны (назовём их Алиса и Боб) заранее договариваются о двух **общедоступных параметрах**:
  * большое простое число p;
  * первообразный корень g по модулю p.
    Эти значения могут быть известны всем, включая потенциального злоумышленника.

2. Каждая сторона генерирует **свой секретный ключ**:
  * Алиса выбирает случайное число a и держит его в секрете;
  * Боб выбирает случайное число b и держит его в секрете.

3. Стороны вычисляют **открытые ключи** и обмениваются ими по открытому каналу:
  * Алиса вычисляет A = g^a mod p и отправляет Бобу;
  * Боб вычисляет $B = g^b mod p и отправляет Алисе.

4. Каждая сторона вычисляет **общий секретный ключ**, используя полученный открытый ключ и свой секретный ключ:
  * Алиса: K = B^a mod p = (g^b)^a mod p = g^ab mod p;
  * Боб: K = A^b mod p = (g^a)^b mod p = g^ab mod p.

5. В результате Алиса и Боб получают **одинаковый ключ** K, который можно использовать для симметричного шифрования (например, AES).


Злоумышленник, перехвативший p, g, A и B, не может легко вычислить K. Для этого ему нужно решить **задачу дискретного логарифмирования** — 
найти a или b по известным A или B. При достаточно больших значениях p эта задача считается вычислительно неразрешимой 
за разумное время.

* Алгоритм Диффи‑Хеллмана **не обеспечивает аутентификацию** сторон. Без дополнительных мер он уязвим к атаке «человек посередине»
(Man‑in‑the‑Middle): злоумышленник может подменить открытые ключи и установить отдельные ключи с Алисой и Бобом. Поэтому
DH обычно используют в сочетании с цифровыми подписями или сертификатами.
* На практике используются усовершенствованные версии алгоритма (например, ECDH — Elliptic Curve Diffie‑Hellman), которые
обеспечивают тот же уровень безопасности при меньших размерах ключей.

## 10. Какие из перечисленных ниже свойств образуют триаду безопасности CIA:</br> а) конфиденциальность;</br> б) неотказуемость;</br> в) аутентичность;</br> г) владение;</br> д) целостность;</br> е) полезность;</br> ж) захват привилегий;</br> з) доступность.

Ответ а), д), з)

1. **Конфиденциальность (Confidentiality)** — гарантия того, что информация доступна только тем, у кого есть соответствующие права доступа.
    * **Цель:** предотвратить несанкционированное ознакомление с данными.
    * **Примеры защиты:** шифрование данных, разграничение прав доступа, аутентификация.
    * **Пример нарушения:** утечка персональных данных из базы данных.

2. **Целостность (Integrity)** — гарантия того, что данные не были изменены или повреждены несанкционированно. Информация остаётся точной и достоверной.
    * **Цель:** защитить данные от несанкционированных изменений (случайных или преднамеренных).
    * **Примеры защиты:** контрольные суммы (хеши), цифровые подписи, журналы аудита.
    * **Пример нарушения:** злоумышленник изменил сумму в платёжном поручении.

3. **Доступность (Availability)** — гарантия того, что информация и ресурсы доступны авторизованным пользователям тогда, когда это необходимо.
    * **Цель:** обеспечить бесперебойную работу систем и доступ к данным.
    * **Примеры защиты:** резервное копирование, отказоустойчивость, защита от DDoS‑атак.
    * **Пример нарушения:** веб‑сайт компании недоступен из‑за DDoS‑атаки.

**Почему не остальные варианты:**

* **б) Неотказуемость (Non‑Repudiation)** — возможность доказать факт выполнения действия конкретным лицом, чтобы оно не
могло от него отказаться. Важное свойство, но не входит в триаду CIA.
* **в) Аутентичность (Authenticity)** — подтверждение подлинности субъекта или данных (например, проверка, что письмо 
действительно пришло от заявленного отправителя). Часто связана с аутентификацией и цифровой подписью. Входит в расширенные
модели безопасности, но не в базовую триаду.
* **г) Владение** — не является стандартным термином в информационной безопасности в контексте модели CIA.
* **е) Полезность** — субъективная характеристика данных, не относится к фундаментальным свойствам безопасности.
* **ж) Захват привилегий (Privilege Escalation)** — это тип атаки, при котором злоумышленник получает повышенные права доступа,
а не свойство безопасности.

Триада CIA — фундаментальная модель в информационной безопасности. Она задаёт три ключевых требования к защите информации:
* **C**onfidentiality (конфиденциальность) — «кто может читать»;
* **I**ntegrity (целостность) — «не изменено ли»;
* **A**vailability (доступность) — «можно ли получить сейчас».

## 11. Укажите, какие из перечисленных атак являются активными (выберите вариант ответа):</br> а) прослушивание сетевого трафика;</br> б) спуфинг;</br> в) социальный инжиниринг;</br> г) атака «отказ в обслуживании»;</br> д) внедрение вируса.

Ответ б), г), д)

**Активные атаки** предполагают вмешательство в работу сети или системы: злоумышленник не просто наблюдает, а изменяет,
прерывает или подменяет данные, воздействует на ресурсы. 

* **б) Спуфинг** (Spoofing) — злоумышленник маскируется под доверенный источник, подменяя IP‑адрес, MAC‑адрес, email и т.д.
Это прямое вмешательство: он активно выдаёт себя за другого, чтобы обмануть систему или пользователя.
* **г) Атака «отказ в обслуживании»** (DoS/DDoS) — злоумышленник перегружает ресурс (сервер, сеть) огромным количеством 
запросов, чтобы сделать его недоступным для легитимных пользователей. Действия целенаправленны и активны.
* **д) Внедрение вируса** — злоумышленник активно помещает вредоносное ПО в систему. Вирус может повреждать данные, красть
информацию, давать удалённый доступ и т.п. — это прямое воздействие на целевую систему.

**Почему не остальные варианты:**

* **а) Прослушивание сетевого трафика** (сниффинг) — **пассивная атака**. Злоумышленник просто перехватывает и анализирует
трафик, не вмешиваясь в его передачу и не изменяя данные.
* **в) Социальный инжиниринг** — метод манипуляции людьми для получения конфиденциальной информации (паролей, данных карт и т.д.).
Хотя он требует действий от злоумышленника, это скорее **психологическая тактика**, а не техническая атака на сеть или систему.
В классификации сетевых атак его обычно не относят к активным в том же смысле, что спуфинг или DoS.
* **е) Полезность** — не является видом атаки. Это характеристика данных или ресурса (насколько они ценны и применимы для пользователя).
* **ж) Захват привилегий** (Privilege Escalation) — это **результат** успешной атаки (например, после внедрения вируса 
или эксплуатации уязвимости), а не самостоятельный тип атаки. Сам по себе он не описывает действие злоумышленника в сети.
* **з) Доступность** — не атака, а одно из свойств информационной безопасности (часть триады CIA). Означает, что ресурсы 
должны быть доступны авторизованным пользователям.

## 12. Укажите, какие из следующих утверждений являются ошибочными:</br> а) Нет смысла предусматривать подсистему аутентификации приложения, основанную на использовании многоразовых паролей, если многоразовые пароли уже используются и при логическом входе в систему;</br> б) Надежная система авторизации может компенсировать недостаточную стойкость паролей пользователей, которые они использовали при входе;</br> в) Стойкость системы защиты считается достаточной, если затраты на нее превысили запланированный уровень;</br> г) Затраты на обеспечение безопасности информации, по крайней мере, не должны превышать величину потенциального ущерба от ее утраты;</br> д) Иногда стойкость системы обеспечения безопасности считают достаточной, если стоимость ее преодоления злоумышленниками превосходит стоимость полученной ими выгоды.

Ответ а), б), в)

**а) Ошибочное утверждение.**

* **Почему неверно:** даже если многоразовые пароли используются при входе в систему, дополнительная аутентификация на 
уровне приложения может усилить безопасность. Например:
    * злоумышленник, получивший доступ к учётной записи ОС, не получит автоматически доступ к приложению;
    * для особо чувствительных приложений (банковские системы, CRM с персональными данными) оправдан дополнительный барьер.
* **Вывод:** многоуровневая аутентификация повышает общую защищённость, поэтому смысл в такой подсистеме есть.

**б) Ошибочное утверждение.**

* **Почему неверно:** авторизация определяет, *что* пользователь может делать после входа, но не защищает от несанкционированного
входа. Если пароли слабые:
    * их легко подобрать (брутфорс);
    * они могут быть украдены (фишинг, утечки баз данных).
* **Пример:** даже самая строгая система авторизации не поможет, если злоумышленник вошёл в систему под учётной записью
администратора с простым паролем `12345`.
* **Вывод:** авторизация не компенсирует недостатки аутентификации — это разные этапы защиты.

**в) Ошибочное утверждение.**

* **Почему неверно:** затраты на защиту — лишь один из факторов. Достаточная стойкость определяется соотношением:
    * стоимости преодоления защиты для злоумышленника;
    * ценности защищаемой информации;
    * потенциального ущерба от её утечки/компрометации.
* **Пример:** тратить 1 млн руб. на защиту данных, ущерб от утечки которых — 10 тыс. руб., нерационально. И наоборот: защита
критически важных данных должна быть максимально надёжной, даже если это дорого.
* **Вывод:** критерий «затраты превысили запланированный уровень» не говорит о реальной стойкости системы.

**Почему остальные утверждения верны:**

**г) Верное утверждение.**

* Логика проста: если затраты на защиту превышают потенциальный ущерб, такая защита экономически нецелесообразна. Например,
нет смысла тратить миллионы на защиту общедоступной информации.

**д) Верное утверждение.**

* Это **основной экономический принцип информационной безопасности:** защита считается достаточной, если злоумышленнику 
невыгодно её взламывать. Если затраты на взлом (время, ресурсы, риск) выше, чем выгода от украденных данных, атака маловероятна.

## 13. Что из перечисленного может быть секретным ключом криптосистемы (выберите вариант ответа):</br> а) геометрическая фигура;</br> б) сборник детских сказок;</br> в) картина;</br> г) число 5.

Ответ а), б), в), г)

Секретный ключ в криптографии — это **набор данных произвольной природы**, который используется алгоритмом шифрования и 
дешифрования. Главное требование — ключ должен быть:
* известен только уполномоченным сторонам;
* достаточно сложен для угадывания/перебора;
* корректно обрабатываем алгоритмом шифрования.

Разберём каждый вариант:

**а) Геометрическая фигура**
* Может быть ключом, если её форма, размеры и координаты закодированы в числовую или символьную последовательность.
* Пример: координаты вершин многоугольника преобразуются в битовую строку — она и будет ключом.

**б) Сборник детских сказок**
* Весь текст книги или его часть (например, определённые страницы, строки) можно преобразовать в бинарную последовательность — получится длинный ключ.
* Важное условие: обе стороны должны точно знать, *какой именно* текст использовать (издание, страницы, абзацы).
* Пример: взять первые 1000 символов сборника и вычислить их хеш — результат будет ключом.

**в) Картина**
* Изображение можно представить как массив пикселей с числовыми значениями цвета (RGB). Эти данные преобразуются в длинную
битовую последовательность.
* Дополнительно можно использовать метаданные (размер, дата создания и т.д.).
* Пример: хеш от изображения JPEG-файла может служить ключом.

**г) Число 5**
* Простое число тоже может быть ключом — особенно в учебных примерах или упрощённых алгоритмах.
* В реальных системах используют очень большие числа (сотни и тысячи бит), но принцип тот же.
* Пример: в некоторых симметричных шифрах или в алгоритме Диффи‑Хеллмана небольшие числа демонстрируют логику работы,
хотя для безопасности нужны большие значения.

**Важные нюансы**

* **Формат данных.** Криптоалгоритмы работают с битовыми последовательностями. Любой объект (текст, изображение, фигура)
должен быть преобразован в такой формат.
* **Длина и энтропия.** Для реальной защиты ключ должен иметь достаточную длину и случайность (высокую энтропию). «Число 5»
или маленькая фигура слишком слабы против атак перебора.
* **Согласованность.** Обе стороны (отправитель и получатель) должны иметь *идентичную копию* ключа и понимать, как его
интерпретировать.

## 14. В каких из перечисленных ниже средствах обеспечения безопасности используется шифрование (выберите вариант ответа):</br> а) аутентификация;</br> б) авторизация;</br> в) протокол NAT;</br> г) защищенный канал;</br> д) сетевой экран прикладного уровня;</br> е) фильтрующий маршрутизатор;</br> ж) цифровая подпись.

Ответ а), г), ж)

* **а) Аутентификация.** Шифрование применяется для безопасной передачи учётных данных (логина и пароля) и защиты от перехвата. Например:
    * пароли могут храниться в виде хешей (с применением криптографических функций);
    * при двухфакторной аутентификации токены передаются в зашифрованном виде;
    * протоколы вроде Kerberos используют шифрование для обмена ключами и подтверждения подлинности.

* **г) Защищённый канал.** Шифрование — основа создания защищённых каналов связи. Примеры:
    * TLS/SSL — шифруют трафик между браузером и веб‑сервером;
    * IPsec — создаёт зашифрованные туннели в VPN;
    * цель — обеспечить конфиденциальность и целостность данных при передаче по открытым сетям.

* **ж) Цифровая подпись.** Шифрование (а точнее, криптография с открытым ключом) лежит в основе механизма цифровых подписей:
    * подпись создаётся с помощью приватного ключа отправителя;
    * проверяется с помощью публичного ключа;
    * гарантирует подлинность и целостность документа (если данные изменены, подпись станет недействительной).


**Почему остальные варианты не подходят:**

* **б) Авторизация.** Этот процесс определяет права доступа пользователя после успешной аутентификации. Шифрование здесь
**не используется напрямую** — система сверяет роли и разрешения, но не шифрует данные в рамках самой авторизации.

* **в) Протокол NAT (Network Address Translation).** NAT преобразует IP‑адреса пакетов для маршрутизации трафика между сетями.
Он **не выполняет шифрование** — его задача техническая (сопоставление адресов), а не безопасность.

* **д) Сетевой экран прикладного уровня (Application Layer Firewall).** Такие экраны анализируют трафик на уровне приложений (HTTP, FTP и т.д.),
фильтруют запросы по правилам, блокируют атаки. Но **шифрование не является их функцией** — они работают с уже расшифрованным
трафиком (или пропускают зашифрованный без изменений).

* **е) Фильтрующий маршрутизатор (Packet Filtering Router).** Фильтрует пакеты по IP‑адресам, портам и протоколам на сетевом
уровне (L3). **Не выполняет шифрование** — только разрешает или блокирует трафик на основе ACL (списков контроля доступа).

**Краткий итог:**

Шифрование используется там, где нужно:
* защитить данные от перехвата (**защищённый канал**);
* подтвердить подлинность сторон и безопасно передать учётные данные (**аутентификация**);
* гарантировать неизменность и авторство информации (**цифровая подпись**).

## 15. Какую роль в обеспечении безопасности компьютерной сети играет протокол NAT?

Ответ - скрывает внутреннюю структуру сети

**Как NAT (Network Address Translation) обеспечивает безопасность:**

1. **Маскировка внутренних IP‑адресов.**
    * NAT преобразует приватные (внутренние) IP‑адреса устройств в сети в один или несколько публичных (внешних) адресов.
    * Внешние узлы видят только публичный IP‑адрес маршрутизатора — они не знают, какие устройства и с какими адресами 
   находятся внутри сети.

2. **Ограничение инициации соединений извне.**
    * По умолчанию NAT не пропускает входящие пакеты, которые не являются ответом на запрос из внутренней сети.
    * Злоумышленник не может напрямую подключиться к компьютеру сотрудника (с приватным IP), так как этот адрес «невидим»
   из Интернета.

3. **Сокращение числа точек атаки.**
    * Вместо того чтобы «выставлять» наружу IP‑адреса всех компьютеров, принтеров и серверов, сеть «представлена» одним
   (или несколькими) публичными адресами.
    * Это уменьшает поверхность атаки: злоумышленнику сложнее найти уязвимое устройство.

4. **Естественная изоляция.**
    * Для доступа извне к какому‑либо внутреннему сервису (веб‑серверу, FTP) нужно явно настроить проброс портов (Port Forwarding).
   Без такой настройки внутренняя сеть остаётся «невидимой».

**Конкретные примеры защиты:**

* **Защита от сканирования портов.** Злоумышленник может просканировать публичный адрес роутера, но не увидит открытые
порты всех внутренних устройств — только те, что специально проброшены.
* **Предотвращение прямых атак.** Атака типа «перебор паролей» на компьютер пользователя из Интернета невозможна, если этот
компьютер не имеет прямого публичного IP и не настроен проброс порта для соответствующего сервиса.
* **Сокрытие топологии сети.** Злоумышленнику сложно понять, сколько устройств в сети, как они связаны и какие роли выполняют
(рабочие станции, серверы и т.д.).

**Важные нюансы:**

* NAT **не является полноценным средством защиты** — это в первую очередь механизм экономии IP‑адресов (IPv4).
* Он не заменяет:
    * файрволы (сетевые экраны), которые фильтруют трафик по сложным правилам;
    * системы обнаружения вторжений (IDS/IPS);
    * шифрование (VPN, TLS) для защиты содержимого трафика.
* Некоторые сценарии требуют отключения или обхода NAT (например, для пиринговых игр, VoIP без STUN/TURN), что может снижать уровень защиты.

Основная роль NAT в безопасности — **сокрытие внутренней структуры сети** и ограничение прямого доступа извне. Это создаёт
базовый «щит» за счёт маскировки приватных IP‑адресов и естественной блокировки несанкционированных входящих соединений. 
Однако для комплексной защиты его необходимо сочетать с другими средствами безопасности.

## 16. Вставьте пропущенные слова («открытый» или «закрытый») в следующее предложение: «Сертификат может быть представлен в форме, состоящей из трех частей: во-первых, … части; во-вторых, той же информации, зашифрованной … ключом сертифицирующей организации, в-третьих, части, представляющей собой первые две части, зашифрованные… ключом владельца».

Ответ - сертификат может быть представлен в форме, состоящей из трёх частей: во‑первых, открытой части; во‑вторых, той же
информации, зашифрованной закрытым ключом сертифицирующей организации, в‑третьих, части, представляющей собой первые две
части, зашифрованные открытым ключом владельца
